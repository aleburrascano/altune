SoundCloud is altune's sole source for the unreleased/leaked/underground long tail — no mainstream catalog (Deezer, iTunes, MusicBrainz) indexes it. `SoundCloudAPIAdapter` (`soundcloud.go`) talks directly to SoundCloud's internal `api-v2.soundcloud.com` JSON backend (the same one the website calls) rather than scraping HTML or shelling to yt-dlp for search.

**Auth model.** No user auth — gated only by a public `client_id` query param embedded in the site's JS bundles. Client-id resolution now lives in its own file, `soundcloud_auth.go` (moved out of `soundcloud.go`, same logic and names). `clientIDResolver` (`newClientIDResolver`, `soundcloud_auth.go`) fetches `https://soundcloud.com/`, regexes the `/assets/*.js` bundle URLs (`scAssetURLRe`), scans them from the end for `client_id:"<32 alnum>"` (`scClientIDRe`), and caches the result. Concurrent cold-start callers collapse via `singleflight`. On a 401/403 (`isAuthStatus`, still in `soundcloud.go`), the adapter calls `resolver.invalidate()` and re-resolves once. This is reverse-engineered and against SoundCloud's ToS — accepted per `AIDEV-DECISION` in the file for self-hosted personal/family use.

**Capabilities (`soundcloud.go`).** `Search` dispatches per requested kind: `searchTracks` (paginated via `next_href` up to `scMaxResults=40`), `searchAlbums` (`/search/albums`, typed playlists with `set_type` album/ep/single), `searchArtists` (`/search/users`). `searchAlbums`, `searchArtists`, `GetArtistTopTracks`, and `GetRelatedTracks` share one generic helper, `scFetchList[T any]`: it runs the auth-retry fetch, decodes one JSON page into `struct{ Collection []T }`, and maps each item via the caller's `mapFn` — only the URL, the item type, and the mapper differ per endpoint (the per-endpoint `scAlbumSearchResponse`/`scUserSearchResponse` page-shape types were dropped, since `scFetchList`'s inline anonymous struct replaces them). `Resolve` implements `ports.ArtworkResolver`, wired last in the artwork chain so it only fires when ID-based sources miss (see [artwork-chain](artwork-chain.md)). `GetArtistTopTracks`/`GetArtistAlbums` implement `ports.ArtistContentProvider`, keyed by the numeric SoundCloud user id OR a profile handle — `resolveUserID` resolves a non-numeric handle (as bridged from a MusicBrainz `soundcloud` url-relation, e.g. `che`) to the numeric id via `/resolve` before fetching; a numeric id passes through untouched. `GetArtistAlbums` returns the artist's typed playlists (album/ep/single) PLUS their standalone track uploads as singles: SoundCloud has no release objects, so a genuine single is just an upload not grouped into a playlist. `fetchArtistPlaylists` collects each playlist's member track ids (from the inline `tracks` array); `fetchArtistStandaloneSingles` maps every upload NOT in that set via `mapSoundCloudStandaloneSingle` (a `record_type=single`, one-track, album-kind result), so an EP track never also renders as a top-level single. Without the standalone half, SC-exclusive drops (a rapper's latest SoundCloud single) never reached the discography. Identity resolves two ways: `externalIDsFromRelations` (MB) now bridges a `soundcloud` url-relation — the profile handle — into the cross-provider identity, so SoundCloud joins the id fan-out **verified**; `ResolveArtistID` (`ports.ArtistIDResolver`, name → SC user id via the top `searchArtists` hit) remains the fallback for artists MusicBrainz never bridges. V2 discography ([merge-dedup](../backend/discovery/merge-dedup.md)) only uses the verified path (the name-guess is disabled there to avoid same-name contamination). `GetRelatedTracks` implements `ports.RelatedTracksProvider` via `/tracks/{id}/related`. `GetAlbumTracks` implements `ports.AlbumContentProvider`: a discography entry's id is either a playlist (album/EP) or a track (a standalone single), so `GetAlbumTracks` tries `/playlists/{id}` first (its member tracks) and falls back to `/tracks/{id}` (the single as a one-element tracklist) — without it a SoundCloud-sourced single fell through to the album-tracks service's blind Deezer title search, which returned a *different* Che album's tracks. `ResolvePermalink` turns a pasted SoundCloud URL into a track via `/resolve?url=`, feeding the import/acquisition workflow (see [pipeline](../backend/acquisition/pipeline.md)). Both album and track mappers now capture a release date via `scBestReleaseDate` (first populated of `release_date` → `display_date` → `created_at`): api-v2 albums carry a clean `release_date` while tracks leave it null but always have `display_date`, so the discography and top-tracks both get a year (fed into `SearchResult.ReleaseDate`, which `normalizeAlbumYears` turns into a display year). `mapSoundCloudAPIAlbum` also lifts `track_count` into `SearchResult.TrackCount` — both fields were previously dropped, leaving SC releases blank on year/count. `mapSoundCloudAPITrack` lifts a `publisher_metadata.isrc` into the typed `SearchResult.ISRC` field when present, letting an officially-distributed SoundCloud upload merge with the same recording from Deezer/MusicBrainz; it now also lifts `publisher_metadata.album_title` into `SearchResult.Album` and converts the api-v2 millisecond `duration` into `SearchResult.Duration` (seconds), fields previously left unset on SoundCloud results. `getJSON` now issues its request through the shared `getBytes`/`withHeader` helpers in `providerhttp.go` instead of building the `http.Request` inline — same behavior, shared plumbing with the other providers.

**Rate limits / caveats.** `api-v2` throttles; the codebase discipline is "don't run concurrent evals against it." `client_id` rotates a few times/month — self-healing, not per-request. Public-only reach (private/unlisted tracks never surface). `SoundCloudAdapter` (`soundcloud_ytdlp.go`) is the original yt-dlp-subprocess adapter, track-only, retained as `searchFallback` when `client_id` resolution is down (wired as `NewSoundCloudAdapter()`, passed as the `fallback` argument to `NewSoundCloudAPIAdapter`).
