`MusicBrainzAdapter` (`musicbrainz.go`) hits `musicbrainz.org/ws/2` keylessly, gated only by a required User-Agent (`cfg.MusicBrainzUserAgent`) — a missing UA returns 403. `rateLimit(ctx)` enforces the hard 1 req/sec ceiling via a mutex that reserves a future slot per caller (fixing an earlier bug where concurrent callers could stamp the same baseline and burst together, triggering MB 503s). HTTP fetches themselves go through the provider-wide `getBytes`/`getJSON`/`withHeader` helpers in `providerhttp.go` (shared across all `adapters/providers/*.go`, not MB-specific) instead of each call site hand-rolling `http.NewRequestWithContext`/`client.Do`/`json.Decode` — this refactor touched all four HTTP call sites in the file: `searchKind` calls the package-level `getBytes` directly (it needs the raw body to branch on `kind` before decoding into the matching struct), while `fetchArtistMatches`, `fetchReleaseGroups`, and `LookupAlbumArtist` were collapsed to a one-line `a.getJSON(ctx, u, &body)`. The adapter's own `getJSON(ctx, u, out)` method (musicbrainz.go, doc'd "Used by the enrichment surface") wraps the shared package-level `getJSON` with the MB User-Agent/Accept headers and calls `rateLimit(ctx)` first — the shared helpers themselves do no rate limiting, so every caller is responsible for invoking `rateLimit` before the fetch. (`LookupAlbumArtist`'s decode-failure debug log was also renamed from `mb.lookup_album_artist_decode_error` to `mb.lookup_album_artist_http_error`.) `Search`/`SearchStructured` cover artist/recording/release-group (`searchKind`, `mbEntity`); recording results additionally carry a `featured_artists` extra parsed from multi-artist credits (`extractMBFeatured` over the joinphrase-aware artist-credit list), the recording `length` mapped into the typed `Duration` (ms→s; decoded onto `mbRecording` but previously dropped), and release-groups the `secondary-types` (Compilation/Live/Soundtrack) into `extras["secondary_types"]` alongside the existing `record_type` from `primary-type`. `ResolveArtistIdentity` resolves a name to an `ArtistIdentity` (MBID, disambiguation, birth year, area, type); `ValidateArtistAlbums`/`LookupAlbumArtist` feed the consensus/contamination-check engine (see [merge-dedup](../backend/discovery/merge-dedup.md)). `ReleaseGroupTitles(mbid)` implements `ports.MBDiscographyAnchor`: it returns the artist's release-group titles (reusing the memoized `fetchReleaseGroups`), used by the V2 discography as the identity-verification anchor — a fan-out provider whose catalogue barely overlaps MB's is a mis-bridged same-name artist (a wrong MB url-relation) and is dropped (`docs/discovery-detail-pipeline.md` §7).

The code also adds an `mbMemo` TTL cache (`mbMemoTTL = 6h`, keyed by normalized name / MBID) around identity resolution and release-group fetches — a performance optimization cutting repeated MB round-trips under the 1 req/s ceiling — not mentioned in `docs/providers/musicbrainz.md`. `ListArtistDiscography` (a discography-listing counterpart to `ValidateArtistAlbums`) likewise exists in code without appearing in the doc's capability list; its `mapMBReleaseGroup` carries the release-group `first-release-date` onto the album's `ReleaseDate` (the authoritative year for albums no catalog provider carries) and its `primary-type` (lowercased) onto `extras.record_type` — MB is the discography completeness spine, so both were load-bearing: the map previously parsed the date into the struct but dropped it (blank years, out of chronological order) and never emitted record_type (MB singles/EPs all fell into the detail screen's Albums row).

`musicbrainz_enrichment.go` implements `ports.MetadataEnricher`: `Lookup` fetches `inc=genres+ratings+url-rels` (artist) or `inc=genres+ratings` (release-group), yielding curated genres (`sortedGenres` — vote-count descending, alphabetical ties), `rating`, and (artist only) `externalIDsFromRelations` — parsing url-relations into `external_ids{discogs,wikidata,spotify,deezer,itunes,soundcloud}` for the cross-provider identity bridge (ADR-0011, see [identity-artwork](../backend/discovery/identity-artwork.md)). The `soundcloud` value is the profile *handle* (`soundcloud.com/che` → `che`), not a numeric id, so the SoundCloud adapter resolves handle→id on use; this is what lets an artist's SoundCloud catalogue join the id-verified discography fan-out instead of being name-guessed (see [soundcloud](soundcloud.md)). `ResolveMBID` does a strict normalized-match lookup (title + artist-credit), never fuzzy — no match returns `""` so the service treats it as "nothing to enrich."

Both search and enrichment are config-gated by `cfg.HasMusicBrainz()`. Cover Art Archive (a sibling adapter, `coverartarchive.go`) runs first in the artwork chain, consuming the release-group MBIDs this adapter supplies (see [artwork-chain](artwork-chain.md)). No popularity signal — MB `rating` is sparse/community-driven; Deezer stays the ranking popularity primary.
