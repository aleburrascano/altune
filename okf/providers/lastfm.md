`LastFmAdapter` (`lastfm.go`) calls `ws.audioscrobbler.com/2.0` with a static `api_key` query param (`cfg.LastFMAPIKey`, gated by `cfg.HasLastFM()`) — no rotation/self-heal needed. `SearchTimeout` is 4s (bumped up from a too-tight 1.5s default that produced spurious timeouts). `Search` covers track/album/artist via `*.search` (`parseLastFmResponse`); `GetArtistTopTracks` accepts either an artist name or an MBID (`looksLikeMBID` prefers `mbid=` over `artist=` for identity-safe lookups, avoiding ambiguous-name collisions); `GetArtistAlbums` is capped at `lastfmAlbumsLimit = 50` — deliberately, since beyond ~50-by-playcount `artist.gettopalbums` returns the artist's entire credited-on graph (compilations, "various artists" appearances) rather than real discography, per a prod coverage scan that exploded the album union 21× at `limit=500`. `GetArtistAlbums` also stamps the typed `SearchResult.MBID` field from `al.MBID` — previously left in the untyped extras map (inert), now feeding merge/artwork the same way it does for MusicBrainz's own results. `FetchCharts` implements `ChartProvider` via `chart.gettop{artists,tracks}`. `parseListeners` normalizes `listeners`/`playcount`, which arrive as strings on search/getInfo but numbers on toptracks/getsimilar.

`lastfm_enrichment.go` implements `ports.LastFmEnricher`: a single kind-dispatched `Lookup(kind, artist, title)` calls `artist.getinfo`/`track.getinfo`/`album.getinfo` with `autocorrect=1` (Last.fm fuzzy-matches server-side, so there's no separate id-resolution step like Discogs needs). Returns `listeners`/`playcount` (the listen-based popularity signal), weighted tags (`parseLastFmTags`, capped at `lastfmTagsCap = 6`), similar artists (`parseLastFmSimilarArtists`, capped at `lastfmSimilarCap = 8`, artist kind only — folded from the same call, no separate endpoint), an `mbid` (a second cross-provider bridge seed alongside MusicBrainz's own url-rels), and a cleaned bio (`cleanLastFmBio` strips the trailing "Read more on Last.fm" anchor + HTML, unescapes entities). See [enrichment](../backend/discovery/enrichment.md).

Caveats: not an artwork source — `*.getInfo` images cap at 300px and artist images are frequently a shared placeholder hash (verified on Radiohead), so artwork stays MB-keyed. Similar-tracks (`track.getSimilar`) and tag-as-discovery (`tag.gettopartists`) remain unbuilt per the doc's next steps. Rate limit ~5 req/sec is `[INFERRED]`, not header-exposed. Code matches `docs/providers/lastfm.md` — no material drift found.
