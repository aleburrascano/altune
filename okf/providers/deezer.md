Deezer is altune's popularity primary (`nb_fan`/`rank`) and the only wired provider carrying lyrics. Two access tiers: the public `api.deezer.com` (no auth, no key — every call in `deezer.go` fires a bare `GET`) and the internal `pipe.deezer.com` GraphQL, reached via an anonymous JWT bootstrapped from `auth.deezer.com/login/anonymous` (reverse-engineered, against ToS, accepted for self-hosted personal/family use per the README doctrine).

`DeezerAdapter` (`deezer.go`) implements `Search`/`SearchStructured` across track/album/artist (`searchKind`, `deezerSearchEndpoint`), `Resolve` (`ArtworkResolver`, preferring 1000px `cover_xl`/`picture_xl` over the 500px `_big` fallback, skipping `IsDeezerPlaceholder`), `GetAlbumTracks`/`GetArtistTopTracks`/`GetArtistAlbums` (`Album`/`ArtistContentProvider`), `FetchCharts` (`ChartProvider`), and `FetchTrackISRC`/`FetchFirstTrackID` feeding the identity/consensus engine (see [merge-dedup](../backend/discovery/merge-dedup.md)). Unconditionally wired in `app.go` — no config gate, since the public API needs no key. `mapDeezerResult` now also populates the typed `Album` (title, from the nil-checked `item.Album`) and `Duration` fields on track results — previously these only lived in the untyped `Extras` map (`"album"`/`"duration"`, display-only, invisible to merge/rank). Populating the typed fields wires Deezer tracks into `completenessOf` (merge) and `isLowConfidenceTail` (rank), both of which branch on `Album`/`Duration` the same way they already do on `ProviderRank`/`ISRC`/`MBID`. Track results also set the typed `DeezerAlbumID` field (formatted `item.Album.ID`) instead of the old `extras["deezer_album_id"]`: `FindRelatedService` (`service/find_related.go`) branches on it to decide whether to fetch Deezer album tracks, so the same producer→consumer compile-check applies — a renamed/typo'd Extras key used to fail silently (empty related-tracks) instead of failing the build.

The same struct also implements `ports.DeezerEnricher` (`deezer_enrichment.go`): `ResolveID` maps `(kind, artist, title)` to a Deezer id via structured search; `Lookup` fetches `/track/{id}` (bpm rounded, gain, `explicit_lyrics`) or `/album/{id}` (label, genres capped at `deezerGenresCap = 6`, record_type, UPC). Both detail calls also parse the `contributors` array into `Featured` guest artists via `extractDeezerFeatured` (skips the first "Main" contributor as the primary artist; on collab albums this surfaces the co-primary artists for the album artist line). Display-only, off the ranking path (see [enrichment](../backend/discovery/enrichment.md)).

`DeezerLyricsAdapter` (`deezer_lyrics.go`) implements `ports.LyricsProvider`: `ResolveTrackID` delegates to the public-API adapter's search; `Lookup` POSTs the `SynchronizedLyrics` query to `pipe.deezer.com`, gated by `deezerJWTResolver` — a singleflight-deduped, self-healing anonymous-JWT cache that re-bootstraps once on `401`. A null `lyrics` object or GraphQL error array is a definitive miss (empty + nil, negative-cacheable), never a failure.

Caveats: rate limit ~50 req/5s per IP is `[INFERRED]`, unconfirmed via headers; `bpm` is sparse (render only when `> 0`); lyrics availability is region/catalog-dependent; the JWT response field name (`jwt`) is explicitly `[INFERRED]` in the code comment, unverified against a live-dumped auth response. Code matches `docs/providers/deezer.md` closely — no material drift found.
