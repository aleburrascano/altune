The `tracks` table (defined in `001_baseline.sql`) persists the catalog context's `Track` aggregate root (see [catalog](../backend/catalog.md)). Columns: `id UUID PK` (default `uuid_generate_v4()`), `user_id UUID NOT NULL` (owning user, no FK — cross-context reference by id per `shared.UserId`), `title`/`artist TEXT NOT NULL`, `album TEXT` (nullable, mapped to Go's `""` when absent), `duration_seconds DOUBLE PRECISION`, `added_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `artwork_url TEXT`, `acquisition_status TEXT NOT NULL DEFAULT 'pending'`, `dedup_key TEXT NOT NULL` with `UNIQUE (user_id, dedup_key)`, plus the extended metadata columns (all present in the baseline itself — there is no later migration adding them): `year`, `genre`, `track_number`, `album_artist`, `isrc`, `audio_ref`, `failure_reason`.

The Go domain type (`track.go`) wraps identity in `TrackId` (unexported `uuid.UUID` field) and models `acquisition_status` as the `AcquisitionStatus` enum (`AcquisitionPending`/`Ready`/`Failed`, zero-value `Pending`) via `String()`/`ParseAcquisitionStatus`. The aggregate enforces the `audio_ref ↔ status` invariant through methods (`MarkReady` requires a non-empty `audioRef`; `MarkFailed` requires `reason` and clears `AudioRef`; `RevertToPending` clears both) — never via direct field mutation. `Track` also carries an ordered `FeaturedArtists []domain.FeaturedArtist` of guest ("feat.") credits — not a `tracks` column but two satellite tables (below), loaded and attached by the repository.

`PgxTrackRepository` (`track_repo.go`) implements `ports.TrackRepository`. `Add` runs in a transaction: the insert uses `ON CONFLICT (user_id, dedup_key) DO NOTHING RETURNING id` — a conflict falls back to `GetByDedupKey` so callers get the existing row without a race-prone read-then-write (the deferred rollback disposes the empty tx) — and a fresh insert writes the track's featured-artist links in the same tx so track and credits commit atomically. `SetTrackNumber` is fill-only (`WHERE track_number IS NULL`), so it never clobbers a real value and is idempotent — it backs the client's persist-as-you-browse write-back of album positions for tracks saved before `track_number` was captured, reporting whether a row actually changed. `Delete` runs inside a transaction that first deletes `playlist_tracks` rows referencing the track, then the track itself. A shared `trackScanDest()` helper builds the canonical 17-column scan/`Track` construction reused by joins in `playlist_repo.go` (see [playlists-table](playlists-table.md)). Reads (`GetByID`/`ListForUser`/`GetByDedupKey`/`ListTracksFeaturing`) attach featured credits via `loadFeaturedForTracks` — one batch query for the whole page, avoiding an N+1; playlist-track joins do not load credits.

Featured credits live in two tables (`010_track_featured_artists.sql`, ADR-0019), normalized so "everything featuring X" is an index-clean join. `featured_artists` is the canonical artist — one row per user per identity: `id UUID PK`, `user_id`, optional `mbid TEXT` / `deezer_id BIGINT`, `name`, `norm_name`, and `identity_key TEXT GENERATED ALWAYS AS (COALESCE(mbid, 'dz:' || deezer_id::text, 'name:' || norm_name)) STORED` with `UNIQUE (user_id, identity_key)`. The generated column is deliberately mirrored by the Go value object's `IdentityKey()` (`domain/featured_artist.go`) so Go-side and SQL-side identity agree. `track_featured_artists` is the ordered membership: `PRIMARY KEY (track_id, featured_artist_id)`, `position INTEGER NOT NULL`, both FKs `ON DELETE CASCADE` (so deleting a track needs no manual link cleanup), plus `idx_tfa_featured_artist` for the featuring lookup. Writes go through `featured_artist_repo.go`: `writeTrackFeatured` upserts the canonical artist (`ON CONFLICT (user_id, identity_key) DO UPDATE SET name = EXCLUDED.name`, so display names refresh) and links it `ON CONFLICT DO NOTHING`; `ReplaceFeaturedArtists` verifies track ownership, deletes the track's existing links, and rewrites the set in one transaction — idempotent, used by the featured backfill.
