The `tracks` table (defined in `001_baseline.sql`) persists the catalog context's `Track` aggregate root (see [catalog/track](../backend/catalog/track.md)). Columns: `id UUID PK` (default `uuid_generate_v4()`), `user_id UUID NOT NULL` (owning user, no FK — cross-context reference by id per `shared.UserId`), `title`/`artist TEXT NOT NULL`, `album TEXT` (nullable, mapped to Go's `""` when absent), `duration_seconds DOUBLE PRECISION`, `added_at TIMESTAMPTZ NOT NULL DEFAULT now()`, `artwork_url TEXT`, `acquisition_status TEXT NOT NULL DEFAULT 'pending'`, `dedup_key TEXT NOT NULL` with `UNIQUE (user_id, dedup_key)`, plus the extended metadata columns (all present in the baseline itself — there is no later migration adding them): `year`, `genre`, `track_number`, `album_artist`, `isrc`, `audio_ref`, `failure_reason`.

`002_relationship_indexes.sql` adds two case-insensitive indexes for cross-user relationship queries (entity-relationship search enrichment): `idx_tracks_lower_album ON tracks (lower(album)) WHERE album IS NOT NULL AND album != ''` and `idx_tracks_lower_artist ON tracks (lower(artist))`.

The Go domain type (`track.go`) wraps identity in `TrackId` (unexported `uuid.UUID` field) and models `acquisition_status` as the `AcquisitionStatus` enum (`AcquisitionPending`/`Ready`/`Failed`, zero-value `Pending`) via `String()`/`ParseAcquisitionStatus`. `NewTrack` validates title/artist and returns the domain's typed `*ValidationError` instead of a bare `errors.New` error (part of the catalog-wide coded-error vocabulary — see [catalog/track](../backend/catalog/track.md)). The aggregate enforces the `audio_ref ↔ status` invariant through methods (`MarkReady` requires a non-empty `audioRef`; `MarkFailed` requires `reason` and clears `AudioRef`; `RevertToPending` clears both) — never via direct field mutation. `Track` also carries an ordered `FeaturedArtists []domain.FeaturedArtist` of guest ("feat.") credits — not a `tracks` column but two satellite tables (below), loaded and attached by the repository.

`PgxTrackRepository` (`track_repo.go`) implements `ports.TrackRepository`. `Add` runs in a transaction: the insert uses `ON CONFLICT (user_id, dedup_key) DO NOTHING RETURNING id` — a conflict falls back to `GetByDedupKey` so callers get the existing row without a race-prone read-then-write (the deferred rollback disposes the empty tx) — and a fresh insert writes the track's featured-artist links in the same tx so track and credits commit atomically. `SetTrackNumber` is fill-only (`WHERE track_number IS NULL`), so it never clobbers a real value and is idempotent — it backs the client's persist-as-you-browse write-back of album positions for tracks saved before `track_number` was captured, reporting whether a row actually changed. `Delete` returns `(deleted bool, audioRef *string, err error)`: it collects the ids of every playlist the track belongs to, deletes its `playlist_tracks` rows, calls the shared `renumberPlaylistPositions` helper (also used by `playlist_repo.go`'s `RemoveTrack`, see [playlists-table](playlists-table.md)) once per affected playlist so removal never leaves a position gap, then deletes the track itself with `RETURNING audio_ref` so the caller can clean up the underlying audio blob. Column lists are centralized in two string constants, `trackColumns` (bare) and `trackColumnsPrefixed` (`t.`-qualified, for joins in `playlist_repo.go`/`featured_artist_repo.go`), so a schema change is a single-line edit; a shared `trackScanDest()` helper builds the matching scan-destination/`Track`-construction pair. `GetByID` does **not** load `FeaturedArtists` — every current caller only reads status/audio-ref fields, so the join would be paid on every call and discarded; `ListForUser`/`GetByDedupKey`/`ListTracksFeaturing` still attach featured credits via `loadFeaturedForTracks` (one batch query per page, avoiding an N+1), and `ListForUser`'s total count now rides along as `COUNT(*) OVER ()` on the same query instead of a separate round-trip. `ListByIDs` batch-loads a user's tracks by id set (`WHERE user_id = $1 AND id = ANY($2)`) in one query — the batch read behind audio-URL presigning, which resolves up to 200 ids per request. Its contract: unknown or foreign ids are simply absent from the result, order is not guaranteed, and — like the playlist-track joins — it deliberately does not load featured credits.

Featured credits live in two tables (`010_track_featured_artists.sql`, ADR-0019), normalized so "everything featuring X" is an index-clean join (see [catalog/featured-artists](../backend/catalog/featured-artists.md) for the Go-side value object and services). `featured_artists` is the canonical artist — one row per user per identity: `id UUID PK`, `user_id`, optional `mbid TEXT` / `deezer_id BIGINT`, `name`, `norm_name`, and `identity_key TEXT GENERATED ALWAYS AS (COALESCE(mbid, 'dz:' || deezer_id::text, 'name:' || norm_name)) STORED` with `UNIQUE (user_id, identity_key)`. The generated column is deliberately mirrored by the Go value object's `IdentityKey()` (`domain/featured_artist.go`) so Go-side and SQL-side identity agree. `track_featured_artists` is the ordered membership: `PRIMARY KEY (track_id, featured_artist_id)`, `position INTEGER NOT NULL`, both FKs `ON DELETE CASCADE` (so deleting a track needs no manual link cleanup), plus `idx_tfa_featured_artist` for the featuring lookup. Writes go through `featured_artist_repo.go`: `writeTrackFeatured` upserts the canonical artist (`ON CONFLICT (user_id, identity_key) DO UPDATE SET name = EXCLUDED.name`, so display names refresh) and links it `ON CONFLICT DO NOTHING`; `ReplaceFeaturedArtists` verifies track ownership, deletes the track's existing links, and rewrites the set in one transaction — idempotent, used by the featured backfill.
