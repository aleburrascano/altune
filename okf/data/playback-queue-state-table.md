`playback_queue_state` (migration `003_playback_queue_state.sql`) has `user_id UUID PRIMARY KEY` — exactly one row per user, no history. Columns: `track_ids TEXT[] NOT NULL DEFAULT '{}'`, `current_idx INTEGER NOT NULL DEFAULT 0`, `position_ms BIGINT NOT NULL DEFAULT 0`, `shuffled BOOLEAN NOT NULL DEFAULT FALSE`, `repeat_mode TEXT NOT NULL DEFAULT 'off'`, `source_id TEXT NOT NULL DEFAULT ''`, `updated_at TIMESTAMPTZ NOT NULL DEFAULT NOW()`, plus `natural_order TEXT[] NOT NULL DEFAULT '{}'` (migration `009_playback_queue_natural_order.sql`). `track_ids` stores the queue in *play* order (so `track_ids[current_idx]` is the current track); `natural_order` stores the same ids in their original, pre-shuffle (album/playlist/library) order, so resume can rebuild the exact shuffled sequence AND un-shuffle back to the original order after relaunch. It is purely a client-side reconstruction aid — the server does no logic on it, round-tripping it as an opaque snapshot like `track_ids`/`source_id`; the `DEFAULT '{}'` means existing rows and non-fidelity clients simply carry an empty `natural_order` and the client falls back to treating `track_ids` as the queue (its prior behavior).

The Go domain type `QueueState` (`queue_state.go`, see [playback](../backend/playback.md)) is deliberately anemic toward `catalog`: `TrackIds` is `[]string`, not a `TrackId` value object — an explicit `AIDEV-DECISION` comment states this is reference-by-id across the context seam, avoiding coupling playback to catalog identity for a snapshot the server never reasons about domain-wise. `RepeatMode` is a 3-value enum (`RepeatOff`/`RepeatAll`/`RepeatOne`, zero-value `Off`). A single private constructor `newQueueState` is "the single door every QueueState passes through": it validates `positionMs >= 0`, normalizes nil `trackIds` and `naturalOrder` to `[]string{}`, and forces `currentIdx` in-range (or 0 for an empty queue). Its fields are grouped into a `QueueStateInput` struct rather than passed positionally — the earlier design took a long positional argument list plus a `WithNaturalOrder` functional option, but since `TrackIds` and `NaturalOrder` are both `[]string`, a positional list risked a silent transposition, and every call site always passed `naturalOrder` anyway, so the "optional" ceremony only hid a semantically required field; `QueueStateInput` fixes both at once. Gate violations (and `ParseRepeatMode` on an unknown mode) return `*ValidationError`, which structurally implements `httputil.StatusError` with status 400 (a plain int — the domain stays free of `net/http`), so client-caused bad input renders as 400 at the HTTP boundary instead of a blanket 500. The gate is reused by both `NewQueueState(in QueueStateInput)` (fresh, stamps `UpdatedAt = now`) and `RehydrateQueueState(in QueueStateInput, updatedAt time.Time)` (from storage, preserving stored `UpdatedAt`). `EmptyQueueState` is the canonical "no queue" value, built by routing a zero-value `QueueStateInput{UserId: userId}` through the same gate so it has no exception to the "single door" invariant.

`PgxQueueStateRepository` (`queue_state_repo.go`, implements `ports.QueueStateRepository`) has exactly two operations: `Upsert` (`INSERT ... ON CONFLICT (user_id) DO UPDATE SET ...`, full-row replace, `natural_order` included) and `GetForUser` (selects all columns including `natural_order`, returns `nil, nil` on `pgx.ErrNoRows` — no row is not an error, just "no snapshot yet" — and rehydrates via `domain.RehydrateQueueState(domain.QueueStateInput{...}, updatedAt)` so the domain invariant gate applies on read).
