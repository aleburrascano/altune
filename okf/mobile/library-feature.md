The user's personal collection screen (`docs/superpowers/specs/2026-06-28-library-redesign-design.md`). Model: **spine + lenses** — the Songs and Playlists chips are what the user owns (primary, fetched/mutated), while Albums and Artists are client-derived groupings of saved tracks (lenses, no backend endpoint). `ui/LibraryScreen.tsx` is the orchestrator: it owns the active `LibraryChip` (`'playlists'|'songs'|'albums'|'artists'`, opens on Playlists), a per-chip sort key, search, and the track context menu, and renders header → search → chip bar → sort control → the one active view — no stacked overview, selecting a chip swaps only the content area.

**Data — SSE-patched, not polled**: `hooks/useLibraryHome.ts` fetches all tracks in one page (`limit: 2000`) under query key `['library-home']` with `staleTime: Infinity` — SSE events from `@shared/events` keep the cache coherent by patching it in place (`trackCachePatch.ts`: `patchTrackInCaches`/`upsertTrackInCaches`/`removeTrackFromCaches` write library-home, the legacy `['library']` infinite cache, and every `['playlist']` detail at once), so the query never background-refetches on mount/nav (F15). The old 5s pending poll was degraded to a 60s belt-and-suspenders `refetchInterval` while any track's `acquisition_status` is `pending` (F14) — a missed terminal acquisition event (see [acquisition](../backend/acquisition.md)) still reconciles without the churn. Pull-to-refresh was removed from all four views (`PlaylistsGrid`/`SongsList`/`AlbumsGrid`/`ArtistsGrid` no longer take `onRefresh`); the error screen's Retry button is the manual escape hatch. The playlist queries follow the same pattern: `hooks/usePlaylistActions.ts`, `ui/AddToPlaylistSheet.tsx`, and `ui/PlaylistDetailScreen.tsx` all set `staleTime: Infinity` because rename/remove/reorder events patch them. **Cache-writer invariant**: because `staleTime: Infinity` pins whatever is written, writers must never fabricate a library snapshot from an absent cache — `undefined` means errored (stale-token 401) or never fetched, NOT empty (`{items: []}` is empty); the detail feature's optimistic save insert once seeded a `total: 1` library over an error state and pinned the lie for the session (`features/detail/save-cache.ts`, see [detail-feature](detail-feature.md)). All patch helpers no-op on absent caches. `hooks/useLibraryGrouping.ts` derives `AlbumGroup`/`ArtistGroup` from the flat track list via `@shared/lib/derive-library-groups` — pure client-side grouping, not domain types. `state.ts`'s `_viewForState` is the same pure loading > error > empty > list precedence pattern used across features. Note: `hooks/useLibrary.ts` today contains only pagination helpers (`_nextOffsetFromPage`/`_flattenPages`) — the live `useInfiniteQuery` hook it once exported was retired and has no current caller; only its own test file still uses the pure helpers.

**Search — a filtered library must never look like a missing one**: `hooks/useLibrarySearch.ts` debounce-commits at ≥2 chars and clears the committed query *immediately* whenever the trimmed input drops below the threshold — the old code handled length 0 and ≥2 but not length 1, so deleting down to one character kept the stale query silently filtering the whole library, which users read as "my library is gone" (the 2026-07-14 report; `__tests__/useLibrarySearch.test.ts`). The hook exposes the committed `query` string so `ui/LibraryNoResults.tsx` (rendered by `LibraryScreen` when `search.hasQuery && active.count === 0`) can name the query, state the library is intact, and offer a one-tap clear. The per-view `emptyLabel`s are now the plain "No songs yet"-style copy only — search-empty is handled exclusively by `LibraryNoResults`.

**Mutations are event-driven, not invalidate-driven (F17)**: `hooks/useDeleteTrack.ts` optimistically removes the track from `['library-home']` (with rollback + `Alert.alert` on error) and performs *no* invalidations — the server's own `track_deleted` event echoes back and removes the row from every other cache. `hooks/useRetryAcquisition.ts` likewise dropped its four invalidations: `onMutate` optimistically flips the row to `pending` (clearing `failure_reason`) across all caches via `patchTrackInCaches`, and the re-queued acquisition's SSE events (started → progress → completed/failed) drive the rest.

**Rows and menus**: `ui/LibraryRow.tsx` shows a live download phase caption (`useDownloadPhase` from `@shared/acquisition/downloadStore` + `phaseLabel`) whenever a download is in flight, *regardless* of the cached `acquisition_status` (F7) — so a re-acquired ready/failed track shows progress too, falling back to the plain pending caption otherwise. The artist line renders `withFeaturing(track.artist, track.featured_artists)`. The ⋮ button measures itself with `measureInWindow` and passes a `MenuAnchor` up, because the old centered `ActionSheet` was replaced by an anchored `ContextMenu` (`@shared/ui/primitives/ContextMenu`) in both `LibraryScreen` and `PlaylistDetailScreen`; menu items are ready-gated — Play Next / Add to Queue appear only when `acquisition_status === 'ready'`, plus Add to Playlist / View Details / Remove.

**Playlist detail**: `ui/PlaylistDetailScreen.tsx` (route `/library/playlist/[id]`) fetches one playlist by id and layers optimistic mutations on it — rename and remove-track use full `onMutate`/rollback optimism (snapshot the name / filter the cached track list), while delete is a plain mutate-then-navigate-back (not optimistic) — each with an `Alert.alert` failure surface. Play/shuffle build a playable queue via `buildPlayableQueue` and hand it to `useQueuePlayback` (see [shared-playback](shared-playback.md)).

**Featuring**: `ui/FeaturingScreen.tsx` (routes `/library/featuring` and `/discover/featuring`, both thin re-exports) lists the user's saved tracks crediting a featured artist via `hooks/useTracksFeaturing.ts` (`['library','featuring',key]` where key is mbid, else `dz:<deezer_id>`, else `name:<name>`; `staleTime: 60s`). It navigates within whichever tab stack it lives in (derived from `useSegments`) so back returns to the launching detail screen, and its empty state never dead-ends — it searches discovery for the artist and opens their detail.

**Navigation**: tracks/albums/artists build a `DiscoveryResult` shape and navigate to `/library/detail` through `useLibraryNavigation`, reusing the same detail-handoff seam `discover` uses — the `detail` feature is kind-agnostic about which tab launched it.
