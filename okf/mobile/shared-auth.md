`supabaseClient.ts` is the app's single `createClient` call (ADR-0006), promoted from `features/auth/api/` once 2+ features depended on it (api-client, playback); every consumer — including the auth feature itself — imports it from here directly (the old feature-local re-export is gone). Session persistence is `expo-secure-store`, NOT `AsyncStorage` (Risk-#3 in the auth-integration spec), via an adapter that maps Supabase's storage interface to SecureStore's async API. Two load-bearing details: `keychainAccessible: AFTER_FIRST_UNLOCK` lets the SDK read tokens when the app wakes in the background on a locked device — the default `WHEN_UNLOCKED` fails the auto-refresh tick with "User interaction is not allowed" — and `getItem` catches SecureStore throws and returns `null` (for tokens stored before that change), which the SDK treats as "no session" and re-authenticates. Web falls back to `localStorage` (or a no-op store), config is `persistSession: true`, `autoRefreshToken: true`, `detectSessionInUrl: false`, and credentials come from `EXPO_PUBLIC_SUPABASE_URL` / `EXPO_PUBLIC_SUPABASE_ANON_KEY`. Invariant: this is the only file that value-imports `@supabase/supabase-js`; everything else takes the singleton, and only `useSession.ts` here (plus feature code) may `import type` from the package.

`useSession.ts` (promoted because auth's AuthGate and settings both consume it) subscribes to `supabase.auth.onAuthStateChange` and exposes the discriminated union `SessionState = {status:'loading'}|{status:'signed-in',session}|{status:'signed-out'}`, defined in the same file. It also owns the cache boundary for SDK-initiated identity changes (AIDEV-WARNING in the file): both the initial `getSession` resolution and every auth event flow through one `apply(session)` function that compares the incoming `user.id` against a ref of the last-seen id and, on change, calls `queryClient.clear()` plus `clearSessionExpired()`. This exists because `queryClient.clear()` in `useSignOut` covers only the explicit Settings sign-out — an SDK-emitted `SIGNED_OUT` (refresh failure) or a `setSession` user swap (deep link) bypasses it, leaving user A's cached library readable by user B. Clearing happens on identity CHANGE only, never on every event: `TOKEN_REFRESHED` fires with the same user and must not wipe the cache (asserted in `__tests__/useSession.test.ts`). A separate `seededRef` distinguishes "haven't seen anything yet" from the real identity `null` (signed-out), so the first observation never clears.

`sessionExpired.ts` is the one feedback edge from HTTP 401 back to auth state. The SDK is the sole authority on session validity but only knows what it sees locally — when the backend rejects a token the SDK still holds (server-side revocation, JWKS rotation, clock skew), no `SIGNED_OUT` is emitted and every query 401s forever. `apiFetch` — the single choke point for every JSON /v1 call (see [shared-api-client](shared-api-client.md); playback's `api/audio.ts` routes its JSON calls through it too, keeping raw fetch only impossible for the native player's own stream requests) — calls `markSessionExpired()` on a server 401; `AuthGate` reads `useSessionExpired()` and renders `SessionExpiredNotice` to offer explicit re-auth. It is a module-level `useSyncExternalStore` store rather than React context because the producer (`apiFetch`) lives outside the React tree. AIDEV-DECISION: marking is deliberately NOT acting — no auto-sign-out on 401, because a server-side JWKS/config blip would bounce every user mid-session at once; the user chooses when to re-authenticate (silent session resurrection is out of scope, see docs/specs/auth-hardening/spec.md). `clearSessionExpired()` resets the flag and is called by `useSession` whenever identity changes.

`useSignOut.ts` (promoted because settings and library/ProfileSheet both consume it) drops the Supabase session AND clears the React Query cache in one motion: `supabase.auth.signOut()` then `queryClient.clear()` — the AC#5(b) half, since a stale cached result from user A leaking to user B would be a multi-tenancy bug at the cache layer. The cache is cleared even when `signOut` errors or throws. It returns a typed `SignOutResult` union (`idle|pending|ok|error`), never boolean flags. Invariant: this hook only covers *explicit* sign-outs; SDK-initiated identity changes (refresh-failure `SIGNED_OUT`, deep-link `setSession` user swaps) bypass it entirely, so `useSession` above owns the complementary half, clearing the cache on any user-id change (and never on same-user `TOKEN_REFRESHED`). Together the two guarantee the cache never carries across identities regardless of which side initiated the change.

Tests live in `__tests__/` here, next to what they cover: `supabaseClient.test.ts` (storage-adapter construction), `useSession.test.ts` (state transitions + cache-clear boundary), `useSignOut.test.tsx` (SDK call + cache clear). The session-expired flow is additionally exercised through `features/auth/__tests__/AuthGate.test.tsx` and `SessionExpiredNotice.test.tsx`.
