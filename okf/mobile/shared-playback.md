The playback subsystem separates two concerns: the **Queue** (an ordered play sequence + cursor, client-owned) and the **native player** (actual audio, reached through `PlaybackControls`, see [[playback-feature]]). `queueStore.ts` defines `useQueueStore`, a Zustand store (`QueueState & QueueActions`) holding `tracks: readonly PlaybackTrack[]`, `playOrder: readonly number[]` (an index permutation — never a re-sorted copy of `tracks`), `currentIndex`, `repeatMode: RepeatMode` (`'off' | 'all' | 'one'`), `shuffled`, and `source: QueueSource | null` (`playlist | library | search`, discriminated union). Play-order indirection lets `toggleShuffle` (Fisher-Yates in `fisherYatesShuffle`, pinning the currently-playing track at position 0) and `reorderQueue`/`removeFromQueue` mutate ordering without touching `tracks`. `cycleRepeatMode` advances `repeatMode` through the `REPEAT_CYCLE` map (off→all→one→off) for the repeat-toggle button; `skipToNext`/`skipToPrevious` are a separate concern — they branch on `repeatMode === 'one'`/`'all'` to decide next-track behavior at queue boundaries, without referencing `REPEAT_CYCLE` directly. `syncCurrentIndex` is an idempotent reconciliation hook so native-driven transitions (lock-screen next/prev) can't feed back into another store-driven skip. `orderedQueueTracks` flattens `playOrder` into the exact sequence handed to the native player — native index, play-order position, and store `currentIndex` are kept 1:1 by invariant.

`types.ts` defines `PlaybackTrack` (title/artist/artworkUrl/durationSeconds plus optional `searchId`/`resultSignature` — discovery provenance threaded onto play/skip/completed telemetry events) and `PlaybackSource` (`library` vs `preview`, discriminated union). `PlaybackControls`/`PlaybackContextValue` is the contract a `PlaybackProvider` (defined in the playback feature) must satisfy; `usePlayback()` (`usePlayback.ts`) reads it via `React.use(PlaybackContext)` and throws if no provider is mounted — this is the seam to a native player implementation elsewhere in the tree.

`canPlay.ts` gates playability on `AcquisitionStatus === 'ready'` — the only place this check lives. `playFromList.ts`'s `buildPlayableQueue` filters a `TrackResponse[]` to ready tracks, maps via `toPlaybackTrack`, and computes `startIndex` for a target track id — the standard way a feature (library, playlist) turns API data into a playable queue. `useQueuePlayback.ts` composes `queueStore` + `usePlayback` into four intents (`playFromList`, `playTrack`, `skipToNext/Previous`, `toggleShuffle`) — the facade every feature UI calls instead of touching the store or native controls directly. `previewUrl.ts`'s `getPreviewUrl` extracts a `preview_url` from a discovery result's untyped `extras` map for search/detail previews that bypass the queue.

Consumed by (2+ features, per the promotion rule): library, playlist, and discover/search playback UI. The server-side counterpart of this Queue — a snapshot persisted for resume-on-reopen — is [[playback]].
