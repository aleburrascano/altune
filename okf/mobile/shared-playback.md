The playback subsystem separates two concerns: the **Queue** (an ordered play sequence + cursor, client-owned) and the **native player** (actual audio, reached through `PlaybackControls`, see [playback-feature](playback-feature.md)). `queueStore.ts` defines `useQueueStore`, a Zustand store (`QueueState & QueueActions`) holding `tracks: readonly PlaybackTrack[]`, `playOrder: readonly number[]` (an index permutation — never a re-sorted copy of `tracks`), `currentIndex`, `repeatMode: RepeatMode` (`'off' | 'all' | 'one'`), `shuffled`, `source: QueueSource | null` (`playlist | library | search`, discriminated union), plus two resume-flow fields: `resumePositionMs` (saved playback offset shown on the scrubber until native progress goes live — once live progress is > 0 the provider ignores and retires the seed, so it never fights real playback or reappears at the top of later tracks) and `generation` (a monotonic ownership token bumped only when the queue is *replaced* — `loadQueue`/`restoreQueue`/`clearQueue`/emptying `removeFromQueue` — never on within-queue transitions like skip/enqueue/shuffle; a slow async flow such as resume reads it before its network fetches and bails if a user tap replaced the queue meanwhile; it never rewinds, deliberately, to prevent ABA).

Shuffle is now tail-only: `toggleShuffle` uses `shuffleTail`, which Fisher-Yates-shuffles only the *upcoming* entries after `currentIndex`, leaving the played history and the current track in place — `currentIndex` stays stable and the native player never has to touch the playing track (no re-buffer). Un-shuffling sorts only the upcoming tail back to natural ascending order; history stays as played. `setShuffled` flips the flag *without* reordering — used by resume, where the loaded order already *is* the shuffled play order. `restoreQueue` is the full-fidelity resume entry point: unlike `loadQueue` (which forces identity play order), it accepts an explicit `playOrder` permutation plus `shuffled`, clamping an out-of-range `currentIndex`, so `tracks` stays in natural (album/playlist) order and un-shuffle returns to it — both paths reset `resumePositionMs` to 0 and bump `generation`.

Repeat semantics deliberately mirror the native player (which is what ships — store skip actions only drive the Expo Go stub, see `isExpoGo.ts`): `cycleRepeatMode` advances `repeatMode` through the `REPEAT_CYCLE` map (off→all→one→off); under `'one'`, `skipToNext` *advances* (repeat-one loops on auto-advance only, matching native `RepeatMode.Track`), only `'all'` wraps past the ends, and `hasNext()` returns false on the last track under `'one'`/`'off'` — native rejects `skipToNext` there, and claiming otherwise rendered an enabled Next button that silently did nothing. `enqueue` (append at play-order tail) and `playNext` (insert at `currentIndex + 1`) carry an AIDEV-WARNING: callers MUST mutate the native queue in lockstep (`TrackPlayer.add`), since native queue position == play-order position; the same warning applies to `reorderQueue` (currently unused — drag-to-reorder isn't wired up). `syncCurrentIndex` is an idempotent reconciliation hook so native-driven transitions (lock-screen next/prev) can't feed back into another store-driven skip. `orderedQueueTracks` flattens `playOrder` into the exact sequence handed to the native player — native index, play-order position, and store `currentIndex` are kept 1:1 by invariant.

`types.ts` defines `PlaybackTrack` (title/artist/artworkUrl/durationSeconds, optional `featuredArtists`, plus optional `searchId`/`resultSignature` — discovery provenance threaded onto play/skip/completed telemetry events) and `PlaybackSource` (`library` vs `preview`, discriminated union). `PlaybackControls`/`PlaybackContextValue` is the contract a `PlaybackProvider` (defined in the playback feature) must satisfy — `startQueue` takes `{ autoplay?, startPositionMs? }` options for resume, and the queue-mutation trio `reorderUpcoming` (replace everything after the current track in place — how shuffle avoids re-buffering), `appendToQueue`, and `insertNext` keep native in lockstep with the store; `usePlayback()` (`usePlayback.ts`) reads it via React's `use(PlaybackContext)` and throws if no provider is mounted — this is the seam to a native player implementation elsewhere in the tree.

`canPlay.ts` gates playability on `AcquisitionStatus === 'ready'` — the canonical statement of the rule (`playFromList.ts` and `useQueueResume.ts` inline the same literal test rather than calling it). `playFromList.ts`'s `buildPlayableQueue` filters a `TrackResponse[]` to ready tracks, maps via `toPlaybackTrack`, and computes `startIndex` for a target track id — the standard way a feature (library, playlist) turns API data into a playable queue. `toPlaybackTrack.ts` also exports `currentTrackToPlaybackTrack`, mapping the server-embedded now-playing snapshot from the queue-state response (`QueueStateCurrentTrack`) with the same shape and trackId identity, so the placeholder queued during resume equals the entry the full rehydrate later builds. `useQueuePlayback.ts` composes `queueStore` + `usePlayback` into the intents every feature UI calls instead of touching the store or native controls directly: `playFromList`, `playTrack`, `addToQueue` (store `enqueue` + native `appendToQueue`; empty queue falls back to just playing the track), `playNext` (store `playNext` + native `insertNext`, or append when current is last since RNTP's `insertBeforeIndex` must be in range), `skipToNext/Previous` (purely native — next track is already buffered), and `toggleShuffle` (store toggle, then `reorderUpcoming` with the upcoming slice only — the playing track is never restarted). `constants.ts` defines `RESTART_THRESHOLD_MS` (3000): pressing "previous" past this restarts the track, under it steps back — shared by the in-app FullPlayer button and the lock-screen/remote handler so the two entry points can't drift. `isCurrentlyPlaying.ts` treats `'loading'` as active alongside `'playing'`, so the now-playing highlight stays on the active track through an auto-advance instead of blanking out mid-transition. `previewUrl.ts`'s `getPreviewUrl` extracts a `preview_url` from a discovery result's untyped `extras` map for search/detail previews that bypass the queue.

Consumed by (2+ features, per the promotion rule): library, playlist, and discover/search playback UI. The server-side counterpart of this Queue — a snapshot persisted for resume-on-reopen — is [playback](../backend/playback.md).
