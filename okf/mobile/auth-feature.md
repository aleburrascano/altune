Owns sign-in, sign-up, sign-out, OAuth (Apple/Google), password reset, and session restoration on the mobile client (ADR-0006). The backend's `auth.RequireUserID` dependency (see [auth](../backend/auth.md)) consumes the JWT this feature obtains via Supabase; the mobile app never decodes or manipulates the token itself.

**Session**: session state lives in `@shared/auth/useSession` (see [shared-auth](shared-auth.md)) — promoted out of this feature once settings became its second consumer. It exposes the discriminated union `SessionState = {status:'loading'}|{status:'signed-in',session}|{status:'signed-out'}` and owns the cache boundary for SDK-initiated identity changes; the details (identity-change `queryClient.clear()`, the `seededRef` seed guard, never clearing on same-user `TOKEN_REFRESHED`) are documented in [shared-auth](shared-auth.md).

**Gate**: `ui/AuthGate.tsx` reads the session state and redirects: loading → splash, signed-out outside `(auth)` → `/sign-in`, signed-in inside `(auth)` → `/library`. A `useSegments()[0] === '(auth)'` guard prevents an infinite redirect loop, and a `reset-password` route segment is let through regardless of session so a password-recovery session (which is technically "signed-in") doesn't bounce the user off the set-new-password screen. New branch: when signed-in per the SDK but the backend is rejecting the token (`useSessionExpired()` from `@shared/auth/sessionExpired`, marked by `apiFetch` on a server 401 — see [shared-auth](shared-auth.md)) and the user is outside `(auth)`, the gate renders `ui/SessionExpiredNotice.tsx` instead of children — without it every screen 401s forever with no way out but hunting down Sign Out in Settings. Signed-out users never see the notice (it would swallow the `/sign-in` redirect). The notice tells the user plainly "Nothing has been lost" (the failure mode it replaces looked like data loss) and its "Sign in again" button calls `useSignOut` from `@shared/auth` — a real sign-out (SDK session dropped AND query cache cleared), which flips `useSession` to `signed-out` and lets AuthGate redirect to `/sign-in`.

**Mutations**: `useSignIn`, `useSignUp`, `useResetPassword`, `useUpdatePassword`, `useOAuth` each wrap one Supabase SDK call and return a typed result union (`idle|pending|ok|error`, etc.), never React state fields that could be checked as booleans. Thrown transport failures are classified by the shared `lib/isNetworkError.ts` predicate (one heuristic, four consumers — network gets distinct actionable copy via `lib/errorCopy.ts`, everything else stays generic). Anti-user-enumeration is load-bearing throughout: sign-in collapses every credential failure mode into the single `invalid_credentials` reason (its only other reasons are `network`/`unknown`), sign-up returns `awaiting-confirmation` identically whether the email is new or already registered, and password-reset always resolves `sent` regardless of whether the address has an account.

**Form**: `ui/AuthForm.tsx` is the shared sign-in/sign-up form (screens pass per-mode copy and which extras to render); validation gates submit before any network call (email format always; password policy + confirm on sign-up), the server staying the backstop. Its email/password fields route through the shared `TextField` primitive (see [shared-ui](shared-ui.md)) with `textContentType`/`autoComplete` wired per field — sign-up uses `newPassword`/`new-password`, sign-in `password` — so OS password managers and autofill work, and the password fields expose a show/hide toggle.

**Deep-link spine**: `lib/parseAuthLink.ts` is a pure classifier for `altune://` URLs, whitelisting only `auth/recovery`, `auth/confirm`, `auth/callback` paths (anything else is `ignored`, per `rn-security.md`). `lib/completeAuthIntent.ts` acts on the classified intent: it exchanges `token_hash`/`access_token` or an OAuth `code` for a session via `supabase.auth.verifyOtp`/`setSession`/`exchangeCodeForSession`, then navigates to `/reset-password` for recovery links. `hooks/useAuthDeepLink.ts` subscribes to `Linking` events and feeds URLs through parse→complete; `useOAuth.ts` reuses the same `completeAuthIntent` after a native `WebBrowser.openAuthSessionAsync` round-trip — one code path covers both providers.

**Storage/API boundary**: the Supabase singleton lives at `@shared/auth/supabaseClient` (see [shared-auth](shared-auth.md)) and every file here imports it from there directly — no feature-local re-export. Session storage is wired to `expo-secure-store`, never `AsyncStorage`. Value-level `@supabase/supabase-js` imports are confined to the shared singleton; feature code only uses `import type`.

Key files: `hooks/useSignIn.ts`, `hooks/useSignUp.ts`, `hooks/useOAuth.ts`, `hooks/useAuthDeepLink.ts`, `lib/parseAuthLink.ts`, `lib/completeAuthIntent.ts`, `lib/isNetworkError.ts`, `ui/AuthGate.tsx`, `ui/SessionExpiredNotice.tsx`, `ui/AuthForm.tsx`, `ui/SignInScreen.tsx`.
