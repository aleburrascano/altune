Owns sign-in, sign-up, sign-out, OAuth (Apple/Google), password reset, and session restoration on the mobile client (ADR-0006). The backend's `auth.RequireUserID` dependency (see [auth](../backend/auth.md)) consumes the JWT this feature obtains via Supabase; the mobile app never decodes or manipulates the token itself.

**Session**: `hooks/useSession.ts` subscribes to `supabase.auth.onAuthStateChange` and exposes a discriminated union `SessionState = {status:'loading'}|{status:'signed-in',session}|{status:'signed-out'}` (`types.ts`). It also owns the cache boundary for SDK-initiated identity changes (AIDEV-WARNING in the file): both the initial `getSession` resolution and every auth event flow through one `apply(session)` function that compares the incoming `user.id` against a ref of the last-seen id and, on change, calls `queryClient.clear()` plus `clearSessionExpired()`. This exists because `queryClient.clear()` in `@shared/auth/useSignOut` covers only the explicit Settings sign-out — an SDK-emitted `SIGNED_OUT` (refresh failure) or a `setSession` user swap (deep link) bypasses it, leaving user A's cached library readable by user B. Clearing happens on identity CHANGE only, never on every event: `TOKEN_REFRESHED` fires with the same user and must not wipe the cache (asserted in `__tests__/useSession.test.ts`). A separate `seededRef` distinguishes "haven't seen anything yet" from the real identity `null` (signed-out), so the first observation never clears.

**Gate**: `ui/AuthGate.tsx` reads the session state and redirects: loading → splash, signed-out outside `(auth)` → `/sign-in`, signed-in inside `(auth)` → `/library`. A `useSegments()[0] === '(auth)'` guard prevents an infinite redirect loop, and a `reset-password` route segment is let through regardless of session so a password-recovery session (which is technically "signed-in") doesn't bounce the user off the set-new-password screen. New branch: when signed-in per the SDK but the backend is rejecting the token (`useSessionExpired()` from `@shared/auth/sessionExpired`, marked by `apiFetch` on a server 401 — see [shared-auth](shared-auth.md)) and the user is outside `(auth)`, the gate renders `ui/SessionExpiredNotice.tsx` instead of children — without it every screen 401s forever with no way out but hunting down Sign Out in Settings. Signed-out users never see the notice (it would swallow the `/sign-in` redirect). The notice tells the user plainly "Nothing has been lost" (the failure mode it replaces looked like data loss) and its "Sign in again" button calls `useSignOut` from `@shared/auth` — a real sign-out (SDK session dropped AND query cache cleared), which flips `useSession` to `signed-out` and lets AuthGate redirect to `/sign-in`.

**Mutations**: `useSignIn`, `useSignUp`, `useResetPassword`, `useUpdatePassword`, `useOAuth` each wrap one Supabase SDK call and return a typed result union (`idle|pending|ok|error`, etc.), never React state fields that could be checked as booleans. Anti-user-enumeration is load-bearing throughout: sign-in collapses every credential failure mode into the single `invalid_credentials` reason (its only other reasons are `network`/`unknown`), sign-up returns `awaiting-confirmation` identically whether the email is new or already registered, and password-reset always resolves `sent` regardless of whether the address has an account.

**Deep-link spine**: `lib/parseAuthLink.ts` is a pure classifier for `altune://` URLs, whitelisting only `auth/recovery`, `auth/confirm`, `auth/callback` paths (anything else is `ignored`, per `rn-security.md`). `hooks/useAuthDeepLink.ts` subscribes to `Linking` events and calls `completeAuthIntent`, which exchanges `token_hash`/`access_token` or an OAuth `code` for a session via `supabase.auth.verifyOtp`/`setSession`/`exchangeCodeForSession`, then navigates to `/reset-password` for recovery links. `useOAuth.ts` reuses this same `completeAuthIntent` after a native `WebBrowser.openAuthSessionAsync` round-trip — one code path covers both providers.

**Storage/API boundary**: `api/supabaseClient.ts` re-exports the singleton from `@shared/auth/supabaseClient` (see [shared-auth](shared-auth.md); promoted there once a second feature needed it); session storage is wired to `expo-secure-store`, never `AsyncStorage`. Value-level `@supabase/supabase-js` imports are confined to the shared singleton; within this feature only `types.ts` and `hooks/useSession.ts` import from the package, and only `import type { Session, ... }`.

Key files: `hooks/useSession.ts`, `hooks/useSignIn.ts`, `hooks/useSignUp.ts`, `hooks/useOAuth.ts`, `hooks/useAuthDeepLink.ts`, `lib/parseAuthLink.ts`, `ui/AuthGate.tsx`, `ui/SessionExpiredNotice.tsx`, `ui/SignInScreen.tsx`.
