Renders the detail screen for a tapped discovery result. It is fed entirely by an in-memory handoff (`@shared/lib/detail-handoff`, written by [discover-feature](discover-feature.md)) — there is no per-item backend fetch on open; an empty handoff (cold start/reload/deep link) redirects to `/discover`. `ui/DetailScreen.tsx` is one vertical scroll: sticky back button → hero (artwork + title + tappable `artist · year` subtitle; for a collab album the subtitle appends Deezer co-primary artists via `withFeaturing` — "Drake, 21 Savage" — while the tap target stays the primary) → a per-kind body (`TrackDetailBody`/`AlbumDetailBody`/`ArtistDetailBody`) → the single remaining `Disclosure`, artist-only ("About <artist>"). There are no tabs, no always-on provider slabs, and no genre pills (`GenrePills` was deleted in the declutter pass — genres surface only inside the artist About).

**Library enrichment (non-destructive)**: library items reach the screen with no `sources` (saved rows store none), so `hooks/useEnrichResult.ts` back-fills them via a discovery search to make the item playable and its tracklist/discography fetchable. The match requires title AND (when known) artist to agree — never title alone, and with no fuzzy fallback — and the merge spreads the stored result last so the library's own `extras` win on every conflict: enrichment may only *add* `sources` plus provider-only keys (mbid, preview_url), never overwrite the album/artist. This fixed the "Green Day by Che" bug where a loose title-only match swapped the real album out from under the user.

**Provider enrichment**: `hooks/useDetailEnrichments.ts` is the single seam deciding which of the three rendered provider hooks (MusicBrainz `useEnrichment` — the header year, all kinds; Deezer — `featured_artists`, track+album; Last.fm — the artist About, artist only) fire for a given `kind`, expressed via each hook's `enabled` flag rather than conditional hook calls (React's rules of hooks). Each provider hook is built on the shared `useEnrichmentQuery` skeleton (Template Method, function-value form) — one `useQuery` with a 24h `staleTime`, gated by a `hasContent` predicate so an unresolved/empty payload collapses to `null` and its section hides. This mirrors the backend's own [enrichment](../backend/discovery/enrichment.md) subsystem. The retired surface was **deleted, not left dark** (2026-07-16 structure audit F1): the Discogs album/artist and lyrics hooks used to fire real fetches for sections that were never rendered post-declutter, and `DiscogsEnrichmentSection`/`DeezerEnrichmentSection`/`DiscogsArtistSection`/`LyricsSection`/`EnrichmentSection` plus their api-client functions are gone (git history keeps them; lyrics belong to the full player when it lands). The surviving detail-open client surface — catalog browse (`getAlbumTracks`/`getArtistTopTracks`/`getArtistAlbums`/`getRelatedTracks`) and the MusicBrainz/Last.fm/Deezer enrichment functions — imports from `@shared/api-client/enrichment`, split out of `discovery.ts` so detail-only growth never churns the module discover depends on (2026-07-17 discover structure audit F1; see [shared-api-client](shared-api-client.md)). The artist About renders only `LastFmEnrichmentSection` — bio (with Read more) → `listeners · plays` → genre chips → similar-artist chips — gated on `enrichments.lastfm !== null`.

**Play**: the track hero action is `ui/PlayButton.tsx`, a 50pt accent circle with play/pause glyph and haptic (feature-local; promotion to shared/ui needs 2+ features). It plays the full Track when in the library (effective id/acquisition status resolved from `extras` OR the live library match — the pure `resolvePlaySource` in `play-source.ts`, unit-tested without rendering), else the 30s Deezer preview (a muted "30s preview" caption sits under it), else disabled — testIDs `detail-play` (library source) / `detail-preview`. Every play is stamped with the handoff's `search_id` and the result's `result_signature` so play/completed events join back to the originating search (the behavioral corpus). Album/artist hero Play is intentionally deferred: their tracklists are discovery results and `buildPlayableQueue` only queues `ready` Tracks, so it would have nothing to play. Featured artists are structured `{name, mbid, deezer_id}` credits (`FeaturedArtist`, parsed by `@shared/lib/featured`'s `featuredArtistsFromExtras`, which tolerates legacy bare-name strings), populated by the backend's FeaturedArtistResolver; the track body's three-tier fallback (structured extras → Deezer enrichment contributors → regex-parsed "feat./ft./with" from title/subtitle) is the pure `resolveFeatured` in `extras.ts`, unit-tested without rendering. The Featuring row navigates to the current tab's `/featuring` browse route carrying the mbid/deezer_id params.

**Save-to-library**: `hooks/useSaveTrack.ts` is a TanStack Query mutation with an optimistic lifecycle over the library snapshot (`libraryKeys.home` from `@shared/lib/query-keys`; the retired `['library']` infinite cache was dropped from the flow by the 2026-07-16 library structure audit) — `onMutate` prepends a pending placeholder (the detail save control and Activity Dock read library-home), `onSuccess` swaps the placeholder for the real server row (`replaceOptimisticTrackHome`) so acquisition SSE events — which carry the real id — match it, deduping by id when the SSE upserted the row first, `onError` rolls back the snapshot, `onSettled` invalidates so a dedup hit reconciles. All transforms are pure in `save-cache.ts`. Invariant: the insert helper never seeds an absent cache — `undefined` means errored/not-fetched, NOT empty (`{items: []}` is the empty library); seeding used to fabricate a one-track library over a 401 error state that `staleTime: Infinity` then pinned for the session (a 273-Track library rendered as one Track/album/artist). The hero save is an outline pill in `TrackDetailBody` (testID `detail-save`) running the acquire lifecycle add → saving → ready/failed from `save-control-state.ts` over the reactive library match, disabled with no POST when the Track has no artist; `TrackSaveControl` remains the 40pt circular row control on album/artist tracklists. The lifecycle's presentation vocabulary is shared, not duplicated: `saveControlLabel`/`saveControlText` live beside the state machine in `save-control-state.ts` and the state→glyph mapping is the single `ui/SaveGlyph.tsx` (the two controls differ only in their `add` glyph props). Reactivity: `hooks/useLibraryTrackMatch.ts` subscribes to the library-home query via a `skipToken` observer (it never fetches, only observes) so an acquisition-SSE `setQueryData` patch re-renders the save control and PlayButton without a library-wide invalidate; the pure lookup `findTrackInData` was split out of the imperative `helpers/find-track-in-library-cache.ts` wrapper for it, and `useIsTrackSaved` is a thin boolean reader. On success the mutation still enqueues a `library_add` telemetry event via the critical outbox (see [shared-telemetry](shared-telemetry.md)), carrying the originating `search_id`.

**Track numbers (derive-then-persist, three layers)**: (1) `toCreateTrackRequest` (`save-cache.ts`) maps `extras.track_position` → `CreateTrackRequest.track_number` (and forwards `featured_artists`), so new album-context saves store the real position. (2) For Tracks saved before that (stored `track_number` null), `_withAlbumPositions` in `hooks/useAlbumDetailState.ts` recovers album order at view time by matching each owned Track against the eagerly-fetched authoritative provider tracklist (provider order IS the album order); a stored position always wins, unmatched Tracks sink to the end, and the derivation is display-only and re-derived per view so it never goes stale. (3) `hooks/usePersistTrackNumbers.ts` fire-and-forgets a `PATCH /v1/tracks/{id}/track-number` per derived position — deduped per screen by a ref, skipping optimistic placeholders — so the database self-heals as albums are browsed; the server write is fill-only (`WHERE track_number IS NULL`), hence idempotent and unable to clobber a real value. The same eager tracklist fetch gates the "More from this album" expander (`detail-more-from-album` hides for singles/fully-owned albums instead of dead-ending) and holds a library album's list until the order is known, so positions are correct on first paint.

**Navigation seams (2026-07-16 structure audit F2/F3)**: `navigation.ts` owns the typed route vocabulary (`TabRoot`/`DetailRoute`, `tabRootFromSegments`, `detailRouteFor`, `featuringRouteFor`) and `openDetail(router, detailRoute, result)` — the only sanctioned way to open a detail screen, pairing the handoff write with the push so the "never push without a handoff" invariant is single-sited (all five former `setDetailHandoff`+`push` call sites route through it). `resolve-entity-query.ts` is the one cache identity for "resolve an entity by name" — a `queryOptions` factory (`['resolve-entity', kind, q, limit]`, 30min staleTime) shared by `useLateralNav` (via `queryClient.fetchQuery`, `retry: false` to keep its fail-fast behavior), `useArtistDiscovery`, `useAlbumDiscovery`, and `useEnrichResult` (limit 5) — so a lateral hop no longer hits the backend twice for the same lookup.

**Lateral navigation**: `useLateralNav` resolves by name through the shared resolve-entity cache and pushes a fresh detail screen for tapped artist/album names, guarded against duplicate pushes. Album/artist content (`useAlbumTracks`, `useArtistContent`) fetches from provider APIs directly (not the handoff), merging MusicBrainz and Deezer sources with an MB-authoritative filter when identity is verified. A library artist's "Your Tracks" list is capped at 5 with a `detail-show-all-tracks` reveal so the discography isn't buried.
