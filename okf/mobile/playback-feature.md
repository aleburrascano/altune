Owns audio playback end-to-end: mini/full player UI, the native queue, lock-screen controls, download-ahead caching, server-side queue resume, and behavioral telemetry. The central design constraint is that `react-native-track-player`'s top-level import touches a native module Expo Go doesn't bundle — importing it unconditionally would crash the app in Expo Go.

**Provider selection** (`hooks/PlaybackProvider.tsx`): a session-constant `isExpoGo` flag chooses which provider implementation to `require(...)` — `expoGoPlaybackProvider.tsx` (no-op, audio inert, `console.warn`s on play) or `trackPlayerProvider.tsx` (the real backend) — using `require` rather than a static `import` specifically so the unused module's native import is never touched. Because the choice is constant for the session, there's no hook-order risk from branching. `expoGoPlaybackProvider` still drives the Zustand queue store (see [shared-playback](shared-playback.md)) on skip so screens remain testable without a dev build.

**TrackPlayerPlaybackProvider** (`hooks/trackPlayerProvider.tsx`) is the real implementation. The public `PlaybackState` (`idle|loading|playing|paused|ended|error`) is now a pure function, `derivePlaybackState.ts`, extracted so the position rules are unit-testable; its buffering branch preserves `positionMs` instead of resetting to 0 — the fix for the resume "flicker" where priming the queue slammed the scrubber to 0:00 before jumping back. Position handling is layered: before native progress goes live (progress 0), the store's `resumePositionMs` seed is displayed so the scrubber lands at the saved offset on relaunch; the moment `livePositionMs > 0` the seed is retired (native progress returns to exactly 0 at the top of *every* track, so a lingering seed would reappear as a jump at each track start). While backgrounded (`useIsForeground`), the display position freezes so the memoized state keeps its identity — the fix for iOS's background-CPU watchdog — and the whole context value is now `useMemo`d so that optimization actually propagates (a bare object literal would re-render every `usePlayback()` consumer twice a second regardless). `seekTo` routes through `seekControls.ts` (`seekPreservingPlayback`): iOS RNTP doesn't reliably re-emit `Playing` after a seek, so `play()` is re-asserted when the track was playing (guarded so scrubbing a paused track never starts audio). `play()` clears the queue store in the same tick as `loadNativeTrack`'s `reset()` so the store never describes a queue the player no longer holds. New best-effort context methods `reorderUpcoming`/`appendToQueue`/`insertNext` mirror store-side shuffle/Add-to-Queue/Play-Next into the native queue; failures are swallowed because the store's play order is already updated and re-syncs on the next queue rebuild.

**Native queue loading, load tokens, and the sync guard** (`loadNativeTrack.ts`, `nativeSyncGuard.ts`): `loadNativeQueue` loads the **entire** ordered queue in one pass so transitions are native and gapless. A load is a multi-await sequence (setup → resolve URLs → reset → add → skip), so every load claims a last-write-wins `loadToken` and bails after each await if a newer load claimed one — the newest caller wins, so a user tap always beats an in-flight resume instead of interleaving `reset()`/`add()` into a mixed queue. Library tracks stream via short-lived presigned URLs minted by `fetchAudioUrls` (`api/audio.ts`, POST `/v1/audio-urls` through `apiFetch` — like `recoverAudio`, so a server 401 marks session-expired; only `audioRequestHeaders` for the native player’s own stream requests stays outside the choke point — with a 2.5s abort so a slow sign never blocks queue start) — direct from storage instead of proxying every byte through the API — capped at `MAX_PRESIGN` (25) from the start index; everything else falls back to the authenticated proxy URL. A single `toNativeTrack` builder (`nativeTrack.ts`) turns a `PlaybackTrack` into the native entry via this presigned-else-proxy ladder (a `streamUrl` option overrides it with a resolved presigned URL or a downloaded local path) — shared by the queue loader here and the download-ahead repair path (`audioPrefetch.ts`), so the mapping lives in exactly one place. `nativeSyncGuard.ts` pins the target index during a load and drops the spurious index-0 `PlaybackActiveTrackChanged` transient that `TrackPlayer.add` emits while priming; it releases on the *first real* index rather than an exact target match, so a missed target can never pin the guard forever and silently drop later transitions. `reorderUpcomingNative` replaces only tracks after the active one (`removeUpcomingTracks` + re-add) so audio continues untouched and native index keeps mirroring store play order; `appendNativeTrack`/`insertNativeTrackNext` do the same for single-track edits.

**Download-ahead cache** (`audioPrefetch.ts`): RNTP's iOS backend does not pre-buffer the next queue item, so every auto-advance stalled ~1s. On each active-track change the *next* library track is downloaded to an `expo-file-system` cache dir and its still-upcoming native entry is swapped to the local path, so the boundary loads from disk with zero buffering. Invariants: only entries strictly **after** the live native active index are ever swapped (RNTP activates the next track when the active one is removed — swapping the playing track skips it, audible as the next button jumping two tracks), and a failed re-add must restore a streaming entry (a missing slot permanently offsets every later native index from the store). Eviction keeps current+4 (`KEEP_WINDOW`) tracks; the `swappedToLocal` set lets the error path distinguish "our cache evicted the file" (repair in place via `TrackPlayer.load`, `repairActiveToStreaming`) from "the server's file is really gone" (`recoverAudio`, POST `/v1/tracks/{id}/audio/recover`, needed because presigned streams bypass the proxy's missing-file recovery). Every path is best-effort: failure leaves the streaming URL playable.

**Service** (`service.ts`, registered outside Expo Go in the root layout): forwards lock-screen remote events to the native player; `RemotePrevious` matches the in-app previous button (restart when past `RESTART_THRESHOLD_MS`, else step back), reading position from the native player because the JS-side position is frozen in background. `PlaybackActiveTrackChanged` is filtered through `shouldApplyActiveIndex` (the sync guard) before mirroring into the store's `syncCurrentIndex`, then kicks `prefetchNext`. `PlaybackError` routes to the repair/recover split above.

**Queue resume** (`hooks/useQueueResume.ts`, pure helpers in `resumeQueue.ts`): the queue persists server-side (`saveQueueState`) every 15s and on backgrounding — track ids in **play order** with `current_index` pointing into that same list, plus `natural_order` (pre-shuffle) so restore rebuilds the exact shuffled sequence *and* can un-shuffle back; `source_id` wire format is `library` / `playlist:<id>:<url-encoded name>` / `search[:<url-encoded query>]` (encoding so a `:` in a name can't be mistaken for a separator). Restore is two-stage: the server embeds the current track's metadata, so a one-track placeholder queue renders now-playing instantly (its store generation is pinned in `placeholderGenerationRef` so the periodic save can never overwrite the real saved queue with it), then the full library is rehydrated via `getTracks` and the play order reconstructed (`reconstructPlayOrder`; non-ready tracks are dropped, so the current track is relocated by id, not raw index — `resolveResumeStartIndex` clamps for legacy rows). Ownership is generation-based: after every await, resume compares the store's `generation` against the one it last wrote and aborts if the user took over — a tap during resume must win, never be silently replaced by yesterday's queue. Finally the native player is primed with the full queue paused (`autoplay: false`) at the saved position, and `setResumePosition` seeds the scrubber until live progress arrives.

**Behavioral signals**: `signals.ts` holds pure helpers — `listenThresholdMs` (30s OR 50% of duration, Kim WSDM 2014) and `buildTrackPayload` — consumed by `usePlaybackSignals` to fire `play`/`completed`/`skip` events without polluting the provider with telemetry logic (see [telemetry](../backend/discovery/telemetry.md) on the backend side). Signals read the raw live position, not the display position, so the resume seed can't spoof the listen threshold.

**UI**: `ui/FullPlayer.tsx` is the fullscreen modal (shuffle and repeat both driven through the `useQueuePlayback` facade so the native queue follows the store, next button disabled when `!hasNext`, scrubber, retry-on-error). `ui/QueueSheet.tsx` shows Now Playing + swipe-to-remove Up Next and drives skip-to / remove / clear-upcoming through the same facade — never the store or native controls directly (that lockstep is the facade's job, see [shared-playback](shared-playback.md)): rows are keyed by `trackIndex` (the entry's slot in `tracks`), not play-order position, because play-order keys shift on every advance and remounted rows inherited swiped-open state; the facade's clear-upcoming reads the store at press time, not from the render closure, because the confirm Alert can sit open across auto-advances and a stale `currentIndex` would delete the playing track. All three players (mini/full/queue) render featured artists via `withFeaturing`. The unified bottom dock — the in-flight downloads bar stacked over `ui/MiniPlayer.tsx` — is composed at the tab shell (`app/(tabs)/_layout.tsx`), the one place that may depend on both the acquisition subsystem and this feature; the download views themselves live with their data in [shared-acquisition](shared-acquisition.md).
