`index.ts` is the base layer: `apiBase` (from `EXPO_PUBLIC_API_URL`, defaulting to `http://127.0.0.1:8000`), the `ApiError` class (`status` + `message`), and `apiFetch<T>(path, init?)` — the single fetch wrapper every typed client function calls. Per ADR-0006, `apiFetch` injects `Authorization: Bearer <access_token>` from the current Supabase session (`supabase.auth.getSession()`, see [auth-feature](auth-feature.md)) — and since every `apiFetch` path is `/v1/*` and requires auth, a missing session now **fails fast**: no network request is sent; `apiFetch` throws `ApiError(401, "API <path> requires a session: …")` before fetch. An AIDEV-NOTE records the trap that motivated this: `getSession` RESOLVES with `{ session: null, error }` on a stale refresh token — it does not throw — so the `error` field must be read explicitly; the old try/catch caught nothing and let the doomed request go out unauthenticated, reporting the resulting 401 as if the server had an opinion. (A `skipAuth` option would be added here if an unauthenticated endpoint ever needed calling — `/health` today is not routed through `apiFetch`.) The complementary edge: when the backend returns 401 on a token the SDK still considers valid (revoked server-side, JWKS rotation, clock skew — cases where the SDK never emits SIGNED_OUT), `apiFetch` calls `markSessionExpired()` from `../auth/sessionExpired` before throwing. Marking is deliberately not acting — no auto-sign-out (a server-side JWKS blip would bounce every user at once); `AuthGate` reads the flag and offers explicit re-auth. Only a server 401 marks it: a 500 does not (backend having a bad day ≠ dead session), and the no-local-session case never reaches the server (the fail-fast fires first, and `AuthGate`'s redirect already covers it) — all pinned by `__tests__/session-expired.test.ts`. `apiFetch` also sets `ngrok-skip-browser-warning` (an AIDEV-NOTE flags this as removable once off ngrok). Non-2xx responses throw `ApiError`; `202`/`204`/`304` short-circuit to `undefined as T` (202 is the audio/recover accepted-with-empty-body case — parsing it as JSON would throw on every success).

Per-context files each export typed functions calling `apiFetch` with a specific path/verb, plus the response/request shapes (hand-maintained, not codegen'd — `types.ts`'s header explicitly flags this as a sync risk mitigated by the plan-reviewer's grep, pending a future OpenAPI codegen spec). `tracks.ts`: `getTracks`, `createTrack`, `deleteTrack`, `retryAcquisition` against `/v1/tracks`, plus `setTrackNumber` (PATCH `/v1/tracks/{id}/track-number` — fill-only on the server, never overwrites; used to write back album positions derived from the album tracklist for tracks saved before `track_number` was captured), `listTracksFeaturing(fa)` ("everything featuring X" over the user's saved tracks, GET `/v1/tracks/featuring` with `mbid`/`deezer_id`/`name` query params, precedence resolved server-side), and `backfillFeaturedArtists` (POST `/v1/tracks/featured-backfill`, returns `{scanned, updated}`). `playlists.ts`: full CRUD + membership (`addTrackToPlaylist`, `reorderPlaylistTracks`) against `/v1/playlists`. `playback.ts`: `getQueueState`/`saveQueueState` against `/v1/playback/queue-state` — the server-side resume-on-reopen snapshot (the memento's caretaker, see [playback](../backend/playback.md)) backing the Queue subsystem (see [shared-playback](shared-playback.md)); `QueueStateResponse` now carries `natural_order` (same tracks as `track_ids` in pre-shuffle album/playlist order, letting restore rebuild the exact shuffled sequence and un-shuffle back; empty for older rows, client falls back to `track_ids` order — `SaveQueueStateRequest` sends it too) and an optional `current_track: QueueStateCurrentTrack`, a display-ready snapshot of the currently-playing track embedded by the server so now-playing renders from this one small call without a full library rehydrate. The discovery surface is two per-context files (split 2026-07-17, discover structure audit F1 — the detail-only surface used to grow inside the module every discover hook imports). `discovery.ts` is the search surface: `searchDiscovery` (normalizes wire `subtitle`/`image_url` `undefined`→`null` via `normalizeResult` since Go's `omitempty` drops empty fields), `suggestDiscovery`, `listSearchHistory`, `clearSearchHistory` (DELETE `/v1/discovery/search-history`), plus the core wire types (`DiscoveryResult`, `DiscoveryKind`, …). The behavioral `/events` send (`recordEvent`) and its `DiscoveryEvent` wire type moved to [shared-telemetry](shared-telemetry.md) (structure audit F1): they are telemetry, not a search call, and keeping them here made the base HTTP layer import *up* into telemetry while telemetry's outbox imported back down — a two-way subsystem dependency. `recordEvent` now calls `apiFetch` from telemetry, so api-client imports nothing from telemetry. `enrichment.ts` is the detail-open surface, consumed only by the detail feature: catalog-browse functions (`getAlbumTracks`, `getArtistTopTracks`, `getArtistAlbums` — now building its query via `URLSearchParams`; the old string concatenation produced a malformed `/albums&name=X` URL whenever `limit` was omitted — and `getRelatedTracks`), and one function per *rendered* detail-enrichment provider (`getEnrichment` for MusicBrainz, `getLastFmEnrichment`, `getDeezerEnrichment`) — each with its own response type documenting "collections always present, unresolved entity returns empty payload" as the null-object contract. The Discogs (`getDiscogsEnrichment`/`getDiscogsArtistEnrichment`) and lyrics (`getLyrics`) clients were deleted with the detail feature's retired enrichment surface (2026-07-16 detail structure audit F1; their sections were never rendered post-declutter); the backend endpoints still exist, so the clients come back from git history if a surface ever renders them. `DeezerEnrichmentResponse` additionally exposes `featured_artists?: unknown[]` — guest ("Featured"-role) contributors from the track detail fetch, wire shape `[{name, mbid?, deezer_id?}]`, parsed via `featuredArtistsFromExtras`.

`types.ts` holds the catalog wire types (`TrackResponse`, `AcquisitionStatus`, `PlaylistResponse`, etc.) consumed by `tracks.ts`/`playlists.ts` and by feature code needing the raw response shape. It also defines `FeaturedArtist` (`{name, mbid: string | null, deezer_id: number | null}` — a guest "feat." credit; ids are present when a provider resolved them and drive tappable navigation + "featuring X" grouping), carried optionally on `TrackResponse.featured_artists` and on `CreateTrackRequest` (alongside the required `track_number: number | null` — 1-based position within its album when saved from an album context, letting the library render real tracklist order instead of counting 1..N).

Consumed by every feature that talks to the backend — discover, library, playlists, playback.
