**Topology** (from `docker-compose.prod.yml` header comments, verified): Postgres is external — Supabase, not containerized here. Redis is self-hosted on the VM (replacing a prior Upstash setup), containerized, not persisted (`--save "" --appendonly no`), capped at `--maxmemory 256mb --maxmemory-policy allkeys-lru`, and NOT published to the host — reachable only by `go-api` over the compose network (`REDIS_URL=redis://redis:6379/0` in `.env.production`). Caddy (`caddy:2-alpine`) is the reverse proxy/TLS terminator, exposing 80/443 to the host, auto-provisioning Let's Encrypt certs, proxying to `go-api:8000` for the domain configured in `Caddyfile` (`altune.duckdns.org`, placeholder to be replaced per-deployment). The `go-api` container builds from the local `Dockerfile`, exposes 8000, reads secrets from `.env.production` (`env_file`), and additionally hardcodes several non-secret values directly in the compose `environment:` block: `OPERATOR_USER_ID` (Mission Control operator account id, see [console-surfaces](../backend/admin/console-surfaces.md) — not secret, since `/admin` still requires that account's Supabase JWT), `EVAL_METER_ENABLED=true` (runs ~5 canonical smoke queries every 6h through a breaker-isolated search instance, can't affect live search), and `SUPABASE_ANON_KEY` (public client key, same as shipped in the app, used so the `/admin` page can sign the operator in). `go-api` depends on Redis being healthy before starting; it mounts a cookies file (`/home/ubuntu/altune/cookies.txt:/data/cookies.txt`, for yt-dlp auth, see [pipeline](../backend/acquisition/pipeline.md)) and a `tmpfs` `/tmp:size=1G`. Caddy's cert state persists in named volumes `caddy-data`/`caddy-config`.

**Deploy workflow** (`.github/workflows/deploy-backend.yml`, see [ci-cd-pipeline](ci-cd-pipeline.md)) — triggers on push to `main` touching `services/go-api/**`, or manual dispatch; serialized via `concurrency.group: deploy-backend` / `cancel-in-progress: false`. It does NOT build anywhere but the VM itself — it SSHes in (`appleboy/ssh-action`, key from repo secret `DEPLOY_SSH_KEY`, host/user from `DEPLOY_HOST`/`DEPLOY_USER`, port defaults 22) and on the VM: fetches + fast-forwards `main` (`git pull --ff-only`), removes any stale renamed container left by an interrupted docker-compose recreate (matches only the `_altune-go-api` suffix so the live container stays serving during rebuild), runs `docker compose -f docker-compose.prod.yml up --build -d`, then `docker image prune -f`. After the SSH step, the workflow itself (not the VM) polls `https://$DEPLOY_HOST/health` up to 5 times / 10s apart and fails the run if 200 is never observed.

**Gotchas**: (1) schema migrations are explicitly NOT applied by this pipeline — the workflow only emits a GitHub Actions warning annotation listing changed files under `services/go-api/migrations/`, expecting the operator to apply them manually via psql before/after the deploy, as appropriate; (2) `command_timeout: 20m` on the SSH step — a build that hangs past 20 minutes fails the deploy without necessarily rolling back the VM's container state; (3) production Redis has zero persistence by design (cache-only, regenerable) — don't expect it to survive a restart, and don't rely on it for anything non-cache.
