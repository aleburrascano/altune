`BackgroundAcquisitionScheduler` (`scheduler.go`) owns job execution; the operator-console job telemetry those jobs produce lives separately in `jobLog` (`joblog.go`). Splitting them means the scheduler's concurrency control and the console's read model can change independently — the scheduler doesn't know how its results are displayed.

**Execution** (`scheduler.go`): `Schedule(userId, trackId, sourceURL)` is fire-and-forget. It first checks `closed` (rejects after `Shutdown`), then dedupes in-flight work per track via `sync.Map.LoadOrStore` on the track id — a second `Schedule` call for a track already running is silently dropped rather than queued or erroring, since the first call will resolve it. Each accepted job runs on its own goroutine, tracked by the shared `*sync.WaitGroup` so `Shutdown` can drain them. Inside the goroutine: a `defer recover()` catches panics, logging the stack and marking the job `failed`/`"panic"` in the log rather than crashing the process; the job then blocks on the semaphore channel (`sem <- struct{}{}`, released via `<-s.sem`) to cap concurrent acquisitions, or bails out as `cancelled` if the scheduler's own lifecycle context (`baseCtx`, distinct from any request context) is done first. Once it has a slot, `s.log.markRunning` flips the job's state, and the pipeline runs via `s.svc.Execute` under a `jobCtx` carrying a `schedulerJobReporter` — the seam that lets `RunPipeline` (see [pipeline](pipeline.md)) push live stage/source updates without the pipeline package knowing anything about scheduling or the console.

**`schedulerJobReporter`** (`scheduler.go`) implements the pipeline's `jobReporter` interface (`job_telemetry.go`): `meta` writes title/artist/album onto the job record once known, `stage` writes the current step name and — if a publisher was wired via `WithSchedulerEvents` — also publishes `track_acquisition_progress` (`track_id`, `stage`) to the user's event stream, and `source` records the resolved candidate URL the first time it's set (a `j.Source == ""` guard, so it doesn't get overwritten by a later step). `WithSchedulerEvents` is scheduler-only — the acquire service's own `WithAcquireEvents` (see [pipeline](pipeline.md)) is a separate publisher for the started/completed/failed events, so a service used outside the scheduler (eval/test paths) still gets its own events without depending on scheduling.

**`jobReporter`** (`job_telemetry.go`) is resolved from context via `jobReporterFrom`, which returns a `noopJobReporter` when none is wired — the pipeline and acquire service call it unconditionally, so eval/test paths that invoke `Execute` directly (no scheduler) are unaffected rather than needing a nil check at every call site.

**`jobLog`** (`joblog.go`) holds three pieces of state behind one mutex: the current queued/running jobs (`map[string]*JobRecord`), a bounded ring of recent terminal outcomes (`recent`, capped at `recentJobCap = 20`, oldest→newest internally but returned newest-first by `snapshot`), and running `succeeded`/`failed` counters (`atomic.Uint64`, incremented in `complete` before the mutex is taken). `complete` is the single call site that advances all three — register a job, run it, then `complete` moves it from `jobs` into `recent` with its outcome and `reason`, computing `ElapsedMs` from the original `ScheduledAt`. There is no separate failure list: failed jobs ride the same `recent` ring as successes, carrying their `reason`, so the operator console reads failures straight off `recent` rather than a parallel structure that could drift out of sync.

`BackgroundAcquisitionScheduler.Status()` combines the scheduler's own `inflightCount` (an `atomic.Int64`, incremented/decremented around each goroutine's lifetime — distinct from `jobLog`'s queued/running map, which tracks identity not just count) with a `jobLog.snapshot()` into `AcquisitionStatus`. All of it — counters, `jobs`, `recent` — is in-memory only and resets on process restart; there is no persistence layer for acquisition job history.

`Shutdown(ctx)` sets `closed` (so no new `Schedule` calls are accepted), cancels `baseCtx` (unblocking any goroutine still waiting on the semaphore or mid-flight, which observe cancellation through the pipeline's own context checks), and waits on the `WaitGroup` with a timeout race against `ctx.Done()` — a slow drain is logged as a warning rather than blocking shutdown indefinitely.
