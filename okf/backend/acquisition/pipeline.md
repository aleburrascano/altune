Acquisition is a customer of the catalog context: it consumes `catalog/domain.Track` and its `MarkReady`/`MarkFailed`/`RevertToPending` invariants rather than owning its own aggregate. There is no `domain/` folder — `AudioCandidate` lives in `ports/ports.go` and doubles as the pipeline's candidate type (same bounded context, same shape — a service-side twin would only duplicate its fields); the pipeline-scoped types (`TrackRef`, `AcquisitionContext`) live in `service/pipeline.go`.

**Pipeline shape** (`service/pipeline.go`): a `Step` interface (`Name`, `Execute`, `Rollback`) run in order by `RunPipeline`, which rolls back completed steps (reverse order, 30s timeout) on failure and wraps the failing step in a `StepError{Step, Err}` — the one structured error shape `failureReason` and the operator console read, rather than parsing an error-message prefix. `AcquireTrackAudioService.buildSteps` (`service/acquire.go`) assembles: `SearchStep` → `SelectStep` → `DownloadStep` → `TagStep` → `StoreStep` → `UpdateTrackStep`. `CoreSteps` (search through store) is factored out separately so the reacquire CLI commands, which update `audio_ref` themselves and stop before the last step, replay the identical sequence instead of a hand-maintained copy that could drift.

- **Search** (`step_search.go`): builds several query variants (ISRC, title+artist, +album, +"audio") and calls `AudioSearcher.Search`, deduping the merged results by URL (`ports.DedupeCandidatesByURL`, shared with the adapter's own engine fan-out below). The yt-dlp adapter (`adapters/ytdlp/searcher.go`) fans each query to two search engines (`ytsearch5:` YouTube, `scsearch5:` SoundCloud — YouTube covers the mainstream catalogue, SoundCloud the unreleased/leaked/underground long tail YouTube doesn't index), merges/dedupes by URL, and only fails if every engine errors.
- **Select** (`step_select.go` + `service/matching.go`): `rankCandidates` scores every candidate with `identityScore` (title/artist fuzzy match via `textnorm.TokenSortRatio`, penalizing title-only matches 0.6x since an unqualified title match is ambiguous — "Die Hard" vs Kendrick's "DIE HARD"), gates out anything below `identityMin` (60), then splits survivors into a Topic-channel bucket (`isTopicChannel`, `- Topic` suffix) and an "other" bucket. Topic candidates sort artist-match first, then feature-match, then identity; other candidates sort by identity, then feature-match (tiebreak), then `metadataRank` (channel/duration/category/view-count blend). Topic-channel candidates always rank ahead of all others — a SoundCloud result only fills a gap when no qualifying YouTube Topic match exists, never displacing a good YouTube one. `extractFeaturedArtists`/`featureMatch` read the *raw* (un-normalized) title because `textnorm.NormalizeForMatch` strips bracketed segments, so by the time `identityScore` compares titles the "(feat. X)" marker is already gone — feature-awareness has to work around that blindness by re-reading the source string.
- **Download** (`step_download.go`): walks the ranked list (capped at `maxVerifyAttempts=4` — each attempt is a full download, so the cap keeps a pathological query from downloading the whole candidate list), downloading each candidate and applying two per-candidate gates before accepting it. Duration gate: when an `AudioProber` is wired and the track has an expected duration, downloads whose probed duration falls outside `durationWithinTolerance` (15s or 7% slack, whichever is larger) are rejected. Decode gate: whenever a prober is wired (independent of expected duration), `ValidateDecodable` must pass — a candidate can carry a valid container and the right duration over corrupt samples that no player can decode (the shipped-m4a defect), so undecodable downloads are discarded and the walk falls through to the next candidate; if every candidate fails, the step errors and the Track is never stored. Both gates fail open when the prober itself errors or times out — verification is skipped rather than blocking acquisition on an unavailable validator. `Rollback` deletes the temp file (`ac.TempPath`, the specific downloaded file, not its containing directory — directory-level cleanup is a separate `CleanupTemp` path in `acquire.go`).
- **Tag** (`step_tag.go` + `adapters/id3/tagger.go`): the step hands the track's metadata to the `AudioTagger` port and swallows (logs) any failure — tagging must never fail the pipeline; without a tagger wired it is a no-op. The id3 adapter writes ID3v2.4 tags (title/artist/album/year/track#/album-artist/genre) via `bogem/id3v2` — but only for `.mp3` files. ID3v2 is an MP3-only convention that prepends a tag block at byte 0; on any other container (e.g. m4a/MP4, which must start with `ftyp` and whose sample-offset table the shift invalidates) that corrupts the file, so the adapter skips non-MP3 files entirely — they carry their metadata in the DB instead.
- **Store** (`step_store.go`): builds a sanitized `userId/artist/album/title.<ext>` audio ref via `buildAudioRef` — the extension is derived from the downloaded temp file itself (`filepath.Ext`, falling back to `.mp3`), so the ref always names what the download actually produced; hardcoding the format here is how refs lied about their bytes during the m4a era — and calls `AudioWriter.Store`; rollback deletes the orphaned object. When a prober is wired (`buildSteps`/`CoreSteps` always pass `WithStoreProber(s.audioProber)`, a no-op unless `WithAudioProber` was configured), it re-runs `ValidateDecodable` on the final temp file before persisting — the last gate after download and tagging, catching corruption introduced by any earlier step (a format-mismatched tagger, a truncated write) before it reaches the library; unlike the tag step, this failure does fail the pipeline. `sanitizePathComponent` also rejects components that normalize to only dots (a path-traversal token) even though the store defends against escapes independently.
- **UpdateTrack** (`step_update_track.go`): calls `Track.MarkReady(audioRef)` + `SetDuration` (from the selected candidate's probed duration, when known) and persists; rollback reverts to pending.

**Ports** (`ports/ports.go`): `AudioSearcher` (search+download), `AudioProber` (`ProbeDuration` for true length plus `ValidateDecodable` for playable samples — metadata-only probing can't catch valid-header/corrupt-data files), `AudioTagger` (`Tag(ctx, path, TrackTags)` — implementations decide per-container applicability, so skipping an unsupported container is not an error; implemented by the id3 adapter), `AudioWriter` (store-side subset of catalog's `AudioStore`: `Exists`/`Store`/`Delete`), `TrackRepository` (narrowed `GetByID`/`Update`). The prober adapter (`adapters/ytdlp/prober.go`, `FfprobeProber`) resolves both ffprobe and ffmpeg binaries near yt-dlp's `--ffmpeg-location` dir (falling back to PATH): duration via ffprobe JSON (30s timeout), decodability via `ffmpeg -v error -i file -f null -` (90s timeout) where a non-zero exit means undecodable (error carries the first stderr line, capped at 200 chars). Both probes fail open — if the binary is missing or times out, verification is skipped rather than blocking acquisition on an unavailable validator.

**Events** (`acquire.go`): the service holds an optional `events.Publisher` (`internal/shared/events`, injected via `WithAcquireEvents`; all publishes are nil-guarded for eval/test paths). `Execute` publishes `track_acquisition_started` (`track_id`) before the pipeline runs — the server-authoritative "it's acquiring now" signal the client seeds its download UI from and uses to flip a re-acquired ready/failed Track back to pending, instead of depending on the optimistic save or the poll — then `track_acquisition_completed` (`track_id`, `audio_ref`) or `track_acquisition_failed` (`track_id`, client-safe `reason`). A per-job `track_acquisition_progress` event (`track_id`, `stage`) is published by the scheduler's per-job reporter as each step transitions — see [scheduling](scheduling.md) for that mechanism, since it isn't reachable through the service alone.

`failureReason` (`acquire.go`) maps internal step errors to a small, stable, client-safe vocabulary ("no matching audio found" for search/select, "audio download failed", "audio storage failed", "audio acquisition cancelled" for context cancellation, generic fallback "audio acquisition failed") by matching on the structured `StepError` — `RunPipeline`'s only other error shape is the "pipeline cancelled" prefix — with the full error chain logged, never stored on the track or returned over the wire.

`AcquireTrackAudioService.Execute` re-searches by metadata on every call; there is no direct-download-by-saved-URL path (SoundCloud's public stream for many tracks is only a ~30s preview, which yt-dlp would otherwise store as if it were the full track). `reconcileForReacquire` decides whether a call should even run the pipeline: a `Ready` track whose audio file still exists is a no-op skip; a `Ready` track with a missing file, or a previously `Failed` track, is reverted to pending and re-run; a fresh `Pending` track proceeds unchanged.
