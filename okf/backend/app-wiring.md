`internal/app` is the composition root: the only place adapters are chosen and injected into ports, per the project rule that the object graph is wired explicitly. `App` (`app.go`) owns config, the pgx pool, the (nilable) Redis client, the HTTP server, and every lifecycle-bound background component (acquisition scheduler, vocab refresh, event bus/feed, alert monitor, log ring, provider health, eval meter). `setup(ctx)` (~30 lines since structure-audit F5, 2026-07-16) assembles in order: pools → auth verifier → event bus → `wireDiscovery` (content/consensus, search service, background tickers, enrichment, handler, vocab refresh) → `wireCatalog` (catalog+acquisition; needs discovery's featured bridge, hence the order) → `wirePlayback` → `mountRoutes` → `wireAdmin` (Mission Control, including `WithMetricsHistory(discoveryPersistence.NewPgxMetricsRollup(a.pool))` for the durable rollup trend panel) → alert monitor; each stage method owns one context's construction and the values crossing contexts travel explicitly through the small `discoveryWiring`/`catalogWiring` structs. `wireDiscovery` itself (`discovery_wiring.go`, split out of `app.go` per structure-audit F1, 2026-07-18) delegates to three sub-stage methods in sequence — `wireDiscoveryConsensus` (the multi-provider album consensus service), `wireDiscoveryContent` (featured-artist bridge + album/artist/related/suggest services, consensus-fed — the album-tracks service resolves native tracklists from Deezer, iTunes, Apple Music, Spotify, and SoundCloud (one shared SoundCloud adapter instance backs the album/artist/related content maps), and any other-sourced album falls back to an artist-guarded Deezer title search), `wireDiscoveryEnrichment` (the MusicBrainz detail-open enrichment service, nil when MB isn't configured) — mirroring the extraction `search_wiring.go` already applied to the search pipeline; each sub-stage returns a small struct or single value, no new interfaces. Event publishers (catalog/acquisition services) receive the bus wrapped in admin's `eventtap.Tap` decorator — the Mission Control system-wide tap — while the per-user SSE handler subscribes to the inner bus directly; the acquisition wiring also hands the acquire service the id3 tagger adapter (`WithAudioTagger`) and gives the scheduler its own tap reference for per-job progress events (`WithSchedulerEvents`) instead of the reporter reaching through the service; the database is *not* nil-tolerant: `database.NewPool` errors on an empty `DATABASE_URL`, so a misconfigured process fails at startup instead of panicking per request. **Nil-tolerant degradation pervades**: nil Redis, nil MusicBrainz, nil audio store, nil enrichment all switch features off rather than fail — e.g. without an audio store the acquisition scheduler is nil and the retry endpoint isn't mounted. Playback's `NowPlayingReader` is deliberately *not* in that set: it is built from the always-present catalog `trackRepo` and passed to `NewQueueService` as a required parameter, so a miswired graph fails at construction instead of silently dropping `current_track` from every resume.

**Detail wiring** (`discovery_wiring.go`): `wireDiscoveryContent` builds the artist-content service over the `artistProviders` map (Deezer, Apple Music, Spotify, SoundCloud, + Last.fm when configured). Apple Music replaces iTunes for artist content — same Apple catalog + ids, but the official Catalog API carries release dates, covers, and ISRC the plain iTunes lookup misses; Spotify joins via its classic `/v1` endpoints with the anonymous web-player token (no persisted-query hash). Each is queried by its bridged id via `providerContentID` (Apple through the shared iTunes id, Last.fm through the MBID) — the "stop excluding them" direction in `docs/brainstorms/2026-07-22-provider-uniform-interface.md`. When a pool exists it also hands that service its own durable `IdentityStore` (`NewRedisIdentityStore(NewPgxIdentityStore(pool), redisClient)` — the same store the search path writes bridges into), and — whenever that store is present — runs the identity-first V2 detail path (the `DETAIL_IDENTITY_FIRST`/`DISCOGRAPHY_V2` flags and their `WithIdentityFirst()`/`WithDiscographyV2()` options were **removed** 2026-07-23; the store's presence is now the sole gate). It reverse-resolves a single `(provider, id)` into the artist's full cross-provider identity and fans discography/top-tracks out by each provider's own id (never a name), so a same-name artist can't bleed in, assembling releases with the rebuilt best-of-merge → confidence-keep → record-type-normalize core (see [merge-dedup](discovery/merge-dedup.md) "Discography V2" and `docs/discovery-detail-pipeline.md` §6); it falls back to the single-provider path whenever the id fan-out comes back empty (or when no store is wired — the unit-test path). The pre-V2 identity path (`identityAlbums`/`identityTopTracks`) was retired at the cutover — the only paths now are V2-identity-first and the single-provider fallback. `WithMBAnchor(sharedMB)` additionally hands V2 the shared MusicBrainz adapter as its identity-verification anchor (`ReleaseGroupTitles`), so a fan-out provider whose catalogue barely overlaps the MBID's release-groups — a mis-bridged same-name artist from a wrong MB url-relation — is dropped (doc §7).

**Search wiring** (`search_wiring.go`): `BuildSearchService` is deliberately the *single construction site* for the search pipeline so the offline eval exercises exactly the ranking users see — the eval CLI, the eval meter, and production all call it. `BuildSearchServiceWithTransport(..., rankingOnly)` adds two seams: an injectable transport (record/replay for fixtures) and `rankingOnly`, which skips display enrichment (artwork chain, related-groups — the bulk of a search's HTTP) and the Redis result cache while keeping every rank-affecting flag live; exploration is *never* wired in rankingOnly so eval order stays deterministic. Eval/synthetic searches always get a nil event store so they never pollute telemetry. Ranking experiment flags: `CROSS_KIND_PROMINENCE_ENABLED` defaults **true** (the only one on by default); `TAIL_DEMOTION`, `EXPLORATION`, `BEHAVIORAL_RANKING`, `EVAL_METER` default false. The durable identity store layers Redis as a read-through front over Postgres (source of truth, survives Redis flushes); `RedisEnrichmentCache` does triple duty as enrichment cache + identity bridge + MBID index, so a detail-open warms the search path. When `IDENTITY_VERIFY_ON_PERSIST` is on, `BuildSearchService` also wires an `IdentityVerifier` (built from the shared MusicBrainz release-group anchor plus deezer/spotify/apple content adapters) so a learned bridge's mis-bridged streaming edges are dropped before the background persist — the permanent identity-fusion fix (see [scatter-gather](discovery/scatter-gather.md)). `BuildSearchServiceWithTransport` takes a pre-built `vocabStore` parameter (falling back to building one from `redisClient` when nil, the offline-tooling path); production passes setup's own instance so **one** vocabulary store now backs both suggest/refresh and search. There remain **two** MusicBrainz adapter instances (one in `wireDiscovery`, one built inside `BuildSearchServiceWithTransport`) — rate safety holds because pacing lives in the shared transport, not the adapter.

**HTTP transport** (`http_client.go`, `live_transport.go`): `defaultLiveTransport` is a package-level `liveTransport` shared process-wide — per-host `rate.Limiter`s (musicbrainz 1/s, itunes 0.5/s, last.fm 5/s, ytmusic 2/s; burst 4 so one query's per-kind fan-out isn't limited against itself) plus retry (3 attempts, 429/5xx, linear backoff). It must stay process-shared: per-client limiters would let N providers each do N× a host's limit — exceeding published limits is what got the prod IP throttled (surfaced as spurious search timeouts). `clientFactory`'s zero value uses the live transport; eval injects record/replay, and replay never touches the live transport. YouTube Music is transport-injected (its library owns a package-global client).

**SSE** (`sse_handler.go`): lives in the composition root deliberately — it is the seam between `shared/events` and the auth context, so `shared/events` never imports auth. Hard-won contracts: never reply 204 to an empty `Last-Event-ID` replay (204 tells EventSource to stop reconnecting — a past hot-loop bug), emit `event: resync` when the ring has evicted events in the requested gap, flush `:ok` immediately, heartbeat every 25s.

**Operator surfaces**: three isolated search stacks exist so failures can't cross-contaminate circuit breakers — production `searchSvc` (breaker inside `NewService`), the eval meter's dedicated `BuildSearchServiceWithTransport(..., rankingOnly)` instance (`eval_runner.go`, off by default per its AIDEV-NOTE; smoke-checks 5 fixed queries against a 0.80 baseline; the meter type itself is `admin/evalmeter.Meter`, its `Runner` implemented here), and Mission Control's re-run (`rerun.go`, builds providers via the exported `BuildDiscoveryProviders` over a recording transport with no Service — bypasses breakers by construction, records per-stage waterfalls via an `ExchangeRecorder`). Since structure-audit F1 (2026-07-16) the re-run ranks with `RankOptions` built from the same config flags production reads plus the live service's `BehavioralScoresSnapshot` — the waterfall runs the identical flag-gated composition, not a zero-value config. It ranks through `discoveryService.RankExplain` (not `RankWith`) so each result's scoring math survives into the response; `projectScored` (`rerun.go`) pairs each `ScoredRow` with its display projection, and `Reshape` still runs over the bare results for the final stage. `search_inspector.go` adapts the *live* service for Mission Control test-search (real pipeline, result cache bypassed, no telemetry). Admin routing is two-layer: shell/config public, all data routes behind auth + `OperatorOnly` (fails closed when `OperatorUserID` is unset). The public `/health` exposes ok/degraded only — dependency topology is operator-gated, and "not_configured" is deliberately distinct from "down".

**Lifecycle** (`Run`): SIGINT/SIGTERM via `signal.NotifyContext`; shutdown order with individual timeouts — server (30s) → alert monitor → event feed → eval meter → vocab refresh (5–10s each) → acquisition scheduler (30s) → a bounded 30s drain of the shared WaitGroup (`drainBackground`, owned by `Run` since structure-audit F4, 2026-07-16 — a hung ticker can no longer wedge shutdown on the no-audio-store path, which used to bare-`wg.Wait()`) → pool/Redis close last. Background tickers (behavioral refresh 30min, nightly corpus materialize when `BEHAVIORAL_CORPUS_PATH` set, metrics rollup 6h — always on, alert monitor 30s) bind to the Run context and register on the shared WaitGroup.

**cmd/ reuse**: `cmd/api` runs `app.New(...).Run` (subcommands like `migrate-dedup`/`backfill-duration` deliberately bypass app with their own CLI audio-store builder). `cmd/discoveryeval` reuses `BuildSearchService`/`BuildSearchServiceWithTransport`/`BuildVocabularyStore`/`BuildConsensusProviders` — see [eval-harness](discovery/eval-harness.md) — plus `BuildArtistContentService` (`detail_harness.go`), the seam its `detail` mode drives: the production artist-content fan-out + MB anchor over a caller-supplied identity store, so the harness can feed deliberately fractured identities and assert the read-time guards clean them (a harness seam only — production still wires the detail service through `wireDiscoveryContent`). `cmd/discoverytrace` reuses the exported `BuildDiscoveryProviders` over its recording transport (its hand-mirror had already drifted — SoundCloud lost the yt-dlp fallback) and ranks through the exported `RankWith`/`Reshape` with the same config flags; it still skips identity-stamping and artwork, and behavioral is nil offline. `buildDiscoveryProviders` (`app.go`) is the one definition both `BuildDiscoveryProviders` and the production search wiring build from — its current roster is Apple Music, Deezer, MusicBrainz (when configured), Last.fm (when configured), SoundCloud (api-v2 with a yt-dlp fallback), YouTube Music, Amazon Music (its internal web-player backend, the same self-hosted-personal-use risk acceptance as SoundCloud's api-v2 client — see `amazonmusic.go`'s AIDEV-DECISION), and Spotify (`spotify.go`, its own anonymous web-visitor bootstrap — a TOTP-gated access token ported from the open-source SpotifyScraper project plus a separate client-integrity token; the most fragile provider in this roster since Spotify actively rotates both the TOTP secrets and the search persisted-query hash, accepted because the other three providers already degrade gracefully around a Spotify outage). Apple Music (`applemusic.go`) replaced plain `ITunesAdapter` in this fan-out (2026-07-22): both hit the identical Apple catalog under the identical ids, but Apple Music's official Catalog API — reached through the anonymous developer token Apple's own web player embeds publicly (`applemusic_token.go`, same risk-acceptance posture) — additionally carries ISRC, composer credits, and a lyrics flag that the plain Search API never exposed, so it strictly supersedes iTunes for fan-out purposes; `ITunesAdapter` itself is untouched and still wired for the artwork chain, album consensus, and content lookups.
