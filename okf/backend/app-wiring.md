`internal/app` is the composition root: the only place adapters are chosen and injected into ports, per the project rule that the object graph is wired explicitly. `App` (`app.go`) owns config, the pgx pool, the (nilable) Redis client, the HTTP server, and every lifecycle-bound background component (acquisition scheduler, vocab refresh, event bus/feed, alert monitor, log ring, provider health, eval meter). `setup(ctx)` (~30 lines since structure-audit F5, 2026-07-16) assembles in order: pools → auth verifier → event bus → `wireDiscovery` (content/consensus, search service, background tickers, enrichment, handler, vocab refresh) → `wireCatalog` (catalog+acquisition; needs discovery's featured bridge, hence the order) → `wirePlayback` → `mountRoutes` → `wireAdmin` (Mission Control, including `WithMetricsHistory(discoveryPersistence.NewPgxMetricsRollup(a.pool))` for the durable rollup trend panel) → alert monitor; each stage method owns one context's construction and the values crossing contexts travel explicitly through the small `discoveryWiring`/`catalogWiring` structs. `wireDiscovery` itself (`discovery_wiring.go`, split out of `app.go` per structure-audit F1, 2026-07-18) delegates to three sub-stage methods in sequence — `wireDiscoveryConsensus` (the multi-provider album consensus service), `wireDiscoveryContent` (featured-artist bridge + album/artist/related/suggest services, consensus-fed), `wireDiscoveryEnrichment` (the MusicBrainz detail-open enrichment service, nil when MB isn't configured) — mirroring the extraction `search_wiring.go` already applied to the search pipeline; each sub-stage returns a small struct or single value, no new interfaces. Event publishers (catalog/acquisition services) receive the bus wrapped in admin's `eventtap.Tap` decorator — the Mission Control system-wide tap — while the per-user SSE handler subscribes to the inner bus directly; the acquisition wiring also hands the acquire service the id3 tagger adapter (`WithAudioTagger`) and gives the scheduler its own tap reference for per-job progress events (`WithSchedulerEvents`) instead of the reporter reaching through the service; the database is *not* nil-tolerant: `database.NewPool` errors on an empty `DATABASE_URL`, so a misconfigured process fails at startup instead of panicking per request. **Nil-tolerant degradation pervades**: nil Redis, nil MusicBrainz, nil audio store, nil enrichment all switch features off rather than fail — e.g. without an audio store the acquisition scheduler is nil and the retry endpoint isn't mounted. Playback's `NowPlayingReader` is deliberately *not* in that set: it is built from the always-present catalog `trackRepo` and passed to `NewQueueService` as a required parameter, so a miswired graph fails at construction instead of silently dropping `current_track` from every resume.

**Search wiring** (`search_wiring.go`): `BuildSearchService` is deliberately the *single construction site* for the search pipeline so the offline eval exercises exactly the ranking users see — the eval CLI, the eval meter, and production all call it. `BuildSearchServiceWithTransport(..., rankingOnly)` adds two seams: an injectable transport (record/replay for fixtures) and `rankingOnly`, which skips display enrichment (artwork chain, related-groups — the bulk of a search's HTTP) and the Redis result cache while keeping every rank-affecting flag live; exploration is *never* wired in rankingOnly so eval order stays deterministic. Eval/synthetic searches always get a nil event store so they never pollute telemetry. Ranking experiment flags: `CROSS_KIND_PROMINENCE_ENABLED` defaults **true** (the only one on by default); `TAIL_DEMOTION`, `EXPLORATION`, `BEHAVIORAL_RANKING`, `EVAL_METER` default false. The durable identity store layers Redis as a read-through front over Postgres (source of truth, survives Redis flushes); `RedisEnrichmentCache` does triple duty as enrichment cache + identity bridge + MBID index, so a detail-open warms the search path. `BuildSearchServiceWithTransport` takes a pre-built `vocabStore` parameter (falling back to building one from `redisClient` when nil, the offline-tooling path); production passes setup's own instance so **one** vocabulary store now backs both suggest/refresh and search. There remain **two** MusicBrainz adapter instances (one in `wireDiscovery`, one built inside `BuildSearchServiceWithTransport`) — rate safety holds because pacing lives in the shared transport, not the adapter.

**HTTP transport** (`http_client.go`, `live_transport.go`): `defaultLiveTransport` is a package-level `liveTransport` shared process-wide — per-host `rate.Limiter`s (musicbrainz 1/s, itunes 0.5/s, last.fm 5/s, ytmusic 2/s; burst 4 so one query's per-kind fan-out isn't limited against itself) plus retry (3 attempts, 429/5xx, linear backoff). It must stay process-shared: per-client limiters would let N providers each do N× a host's limit — exceeding published limits is what got the prod IP throttled (surfaced as spurious search timeouts). `clientFactory`'s zero value uses the live transport; eval injects record/replay, and replay never touches the live transport. YouTube Music is transport-injected (its library owns a package-global client).

**SSE** (`sse_handler.go`): lives in the composition root deliberately — it is the seam between `shared/events` and the auth context, so `shared/events` never imports auth. Hard-won contracts: never reply 204 to an empty `Last-Event-ID` replay (204 tells EventSource to stop reconnecting — a past hot-loop bug), emit `event: resync` when the ring has evicted events in the requested gap, flush `:ok` immediately, heartbeat every 25s.

**Operator surfaces**: three isolated search stacks exist so failures can't cross-contaminate circuit breakers — production `searchSvc` (breaker inside `NewService`), the eval meter's dedicated `BuildSearchServiceWithTransport(..., rankingOnly)` instance (`eval_runner.go`, off by default per its AIDEV-NOTE; smoke-checks 5 fixed queries against a 0.80 baseline; the meter type itself is `admin/evalmeter.Meter`, its `Runner` implemented here), and Mission Control's re-run (`rerun.go`, builds providers via the exported `BuildDiscoveryProviders` over a recording transport with no Service — bypasses breakers by construction, records per-stage waterfalls via an `ExchangeRecorder`). Since structure-audit F1 (2026-07-16) the re-run ranks through `discoveryService.RankWith`/`Reshape` with `RankOptions` built from the same config flags production reads plus the live service's `BehavioralScoresSnapshot` — the waterfall runs the identical flag-gated composition, not a zero-value config. `search_inspector.go` adapts the *live* service for Mission Control test-search (real pipeline, result cache bypassed, no telemetry). Admin routing is two-layer: shell/config public, all data routes behind auth + `OperatorOnly` (fails closed when `OperatorUserID` is unset). The public `/health` exposes ok/degraded only — dependency topology is operator-gated, and "not_configured" is deliberately distinct from "down".

**Lifecycle** (`Run`): SIGINT/SIGTERM via `signal.NotifyContext`; shutdown order with individual timeouts — server (30s) → alert monitor → event feed → eval meter → vocab refresh (5–10s each) → acquisition scheduler (30s) → a bounded 30s drain of the shared WaitGroup (`drainBackground`, owned by `Run` since structure-audit F4, 2026-07-16 — a hung ticker can no longer wedge shutdown on the no-audio-store path, which used to bare-`wg.Wait()`) → pool/Redis close last. Background tickers (behavioral refresh 30min, nightly corpus materialize when `BEHAVIORAL_CORPUS_PATH` set, metrics rollup 6h — always on, alert monitor 30s) bind to the Run context and register on the shared WaitGroup.

**cmd/ reuse**: `cmd/api` runs `app.New(...).Run` (subcommands like `migrate-dedup`/`backfill-duration` deliberately bypass app with their own CLI audio-store builder). `cmd/discoveryeval` reuses `BuildSearchService`/`BuildSearchServiceWithTransport`/`BuildVocabularyStore`/`BuildConsensusProviders` — see [eval-harness](discovery/eval-harness.md). `cmd/discoverytrace` reuses the exported `BuildDiscoveryProviders` over its recording transport (its hand-mirror had already drifted — SoundCloud lost the yt-dlp fallback) and ranks through the exported `RankWith`/`Reshape` with the same config flags; it still skips identity-stamping and artwork, and behavioral is nil offline.
