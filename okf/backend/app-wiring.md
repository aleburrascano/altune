`internal/app` is the composition root: the only place adapters are chosen and injected into ports, per the project rule that the object graph is wired explicitly. `App` (`app.go`) owns config, the pgx pool, the (nilable) Redis client, the HTTP server, and every lifecycle-bound background component (acquisition scheduler, vocab refresh, event bus/feed, alert monitor, log ring, provider health, eval meter). `setup(ctx)` assembles in order: pools → auth verifier → event bus → catalog+acquisition → playback → discovery content/consensus → search service → background tickers → enrichment → discovery handler → vocab refresh → chi router → admin (Mission Control) → alert monitor. **Nil-tolerant degradation pervades**: nil Redis, nil MusicBrainz, nil audio store, nil enrichment all switch features off rather than fail — e.g. without an audio store the acquisition scheduler is nil, the retry endpoint isn't mounted, and shutdown falls back to bare `wg.Wait()`.

**Search wiring** (`search_wiring.go`): `BuildSearchService` is deliberately the *single construction site* for the search pipeline so the offline eval exercises exactly the ranking users see — the eval CLI, the eval meter, and production all call it. `BuildSearchServiceWithTransport(..., rankingOnly)` adds two seams: an injectable transport (record/replay for fixtures) and `rankingOnly`, which skips display enrichment (artwork chain, related-groups — the bulk of a search's HTTP) and the Redis result cache while keeping every rank-affecting flag live; exploration is *never* wired in rankingOnly so eval order stays deterministic. Eval/synthetic searches always get a nil event store so they never pollute telemetry. Ranking experiment flags: `CROSS_KIND_PROMINENCE_ENABLED` defaults **true** (the only one on by default); `TAIL_DEMOTION`, `EXPLORATION`, `BEHAVIORAL_RANKING`, `EVAL_METER` default false. The durable identity store layers Redis as a read-through front over Postgres (source of truth, survives Redis flushes); `RedisEnrichmentCache` does triple duty as enrichment cache + identity bridge + MBID index, so a detail-open warms the search path. There are intentionally **two** vocabulary stores (setup's for suggest/refresh; the search service builds its own) and **two** MusicBrainz adapter instances — rate safety holds because pacing lives in the shared transport, not the adapter.

**HTTP transport** (`http_client.go`, `live_transport.go`): `defaultLiveTransport` is a package-level `liveTransport` shared process-wide — per-host `rate.Limiter`s (musicbrainz 1/s, itunes 0.5/s, last.fm 5/s, ytmusic 2/s; burst 4 so one query's per-kind fan-out isn't limited against itself) plus retry (3 attempts, 429/5xx, linear backoff). It must stay process-shared: per-client limiters would let N providers each do N× a host's limit — exceeding published limits is what got the prod IP throttled (surfaced as spurious search timeouts). `clientFactory`'s zero value uses the live transport; eval injects record/replay, and replay never touches the live transport. YouTube Music is transport-injected (its library owns a package-global client).

**SSE** (`sse_handler.go`): lives in the composition root deliberately — it is the seam between `shared/events` and the auth context, so `shared/events` never imports auth. Hard-won contracts: never reply 204 to an empty `Last-Event-ID` replay (204 tells EventSource to stop reconnecting — a past hot-loop bug), emit `event: resync` when the ring has evicted events in the requested gap, flush `:ok` immediately, heartbeat every 25s.

**Operator surfaces**: three isolated search stacks exist so failures can't cross-contaminate circuit breakers — production `searchSvc` (breaker inside `NewService`), the eval meter's dedicated `BuildSearchService` instance (`eval_runner.go`, off by default per its AIDEV-NOTE; smoke-checks 5 fixed queries against a 0.80 baseline), and Mission Control's re-run (`rerun.go`, builds providers directly with no Service — bypasses breakers by construction, records per-stage waterfalls via an `ExchangeRecorder`). `search_inspector.go` adapts the *live* service for Mission Control test-search (real pipeline, result cache bypassed, no telemetry). Admin routing is two-layer: shell/config public, all data routes behind auth + `OperatorOnly` (fails closed when `OperatorUserID` is unset). The public `/health` exposes ok/degraded only — dependency topology is operator-gated, and "not_configured" is deliberately distinct from "down".

**Lifecycle** (`Run`): SIGINT/SIGTERM via `signal.NotifyContext`; shutdown order with individual timeouts — server (30s) → alert monitor → event feed → eval meter → vocab refresh (5–10s each) → acquisition scheduler (30s, or bare `wg.Wait()`) → pool/Redis close last. Background tickers (behavioral refresh 30min, nightly corpus materialize when `BEHAVIORAL_CORPUS_PATH` set, metrics rollup 6h — always on, alert monitor 30s) bind to the Run context and register on the shared WaitGroup.

**cmd/ reuse**: `cmd/api` runs `app.New(...).Run` (subcommands like `migrate-dedup`/`backfill-duration` deliberately bypass app with their own CLI audio-store builder). `cmd/discoveryeval` reuses `BuildSearchService`/`BuildSearchServiceWithTransport`/`BuildVocabularyStore`/`BuildConsensusProviders` — see [eval-harness](discovery/eval-harness.md). `cmd/discoverytrace` does **not** reuse the builders — its provider set mirrors `buildDiscoveryProviders` by hand (a known drift risk) and skips identity-stamping and artwork.
