`FeaturedArtist` (`domain/featured_artist.go`, ADR-0019) is an immutable value object for a guest ("feat.") credit on a [Track](track.md) ā€” not an entity, no lifecycle beyond track membership. It carries a display `Name` plus optional canonical ids (`MBID`, `DeezerID`); `Role` is always `"featured"` in v1 (`RoleFeatured`). Three constructors cover the credit's three entry points: `NewFeaturedArtist(name, mbid, deezerID)` trims the name and returns `(zero, false)` on empty ā€” used when persisting a credit, so unnamed credits are dropped at construction; `NewFeaturedArtistIdentityOnly` skips the non-empty-name requirement for identity-based lookups where the caller has an mbid/deezer id but no display name; `FeaturedArtistForQuery` (used by the `featuring` query endpoint) tries the full constructor first and falls back to identity-only, so a name-only, id-only, or name+id query param combination all resolve.

`IdentityKey()` ā€” MBID, else `"dz:" + DeezerID`, else `"name:" + NormalizedName()` ā€” is the stable grouping key for "everything featuring X." It deliberately **mirrors** the generated `identity_key` column on the `featured_artists` table (`GENERATED ALWAYS AS (COALESCE(mbid, 'dz:' || deezer_id::text, 'name:' || norm_name)) STORED`, see [tracks-table](../../data/tracks-table.md#featured-credits) for the full column/constraint definitions) so Go-side and SQL-side identity agree ā€” a mismatch between the two precedence chains would silently split one artist's credits across two identity keys. `NormalizedName()` (lowercase, whitespace-collapsed) is both the identity-key fallback and the SQL column's `norm_name` counterpart. See [tracks-table](../../data/tracks-table.md) for the `featured_artists`/`track_featured_artists` table and constraint definitions.

Services:
- **`BackfillFeaturedService`** (`backfill_featured.go`) pages a user's library `backfillPageSize` (200) tracks at a time via a narrow `featuredBackfillRepo` (`ListForUser` + `ReplaceFeaturedArtists`), resolves credits per track through the injected `ports.FeaturedArtistResolver`, and replaces the track's set ā€” idempotent, since `ReplaceFeaturedArtists` is a full delete-then-rewrite, not a merge. A per-track resolver error is logged (`slog.WarnContext`) and skipped so one bad lookup doesn't abort the run; a track with zero resolved credits is left untouched (not cleared).
- **`ListFeaturingService`** (`list_featuring.go`) is a thin pass-through to `ports.TrackRepository.ListTracksFeaturing(userId, fa)` ā€” the "everything featuring X" browse, matched by `fa.IdentityKey()`.

Port (`ports/featured_resolver.go`): `FeaturedArtistResolver.Resolve(ctx, artist, title) ([]domain.FeaturedArtist, error)` resolves credits from the discovery providers. Implemented by `adapters/discoverybridge.FeaturedResolver`, which delegates to `discovery/service.FeaturedArtistResolver` and translates discovery value objects into catalog `FeaturedArtist`s via `domain.NewFeaturedArtist` (dropping unnamed results) ā€” mirrors `playback/catalogbridge`'s pattern, keeping catalog free of a direct discovery import.

Persistence (`adapters/persistence/featured_artist_repo.go`, tables detailed in [tracks-table](../../data/tracks-table.md)): `writeTrackFeatured` upserts each credit into the canonical `featured_artists` table on `ON CONFLICT (user_id, identity_key) DO UPDATE SET name = EXCLUDED.name` (so display names refresh on re-resolve) and links it into `track_featured_artists` in position order (`ON CONFLICT DO NOTHING`) ā€” must run inside a transaction; it does not clear existing links, so replace-semantics callers (the backfill) delete first. `loadFeaturedForTracks` batch-attaches credits for a whole page of tracks in one query (`WHERE tfa.track_id = ANY($1) ORDER BY tfa.track_id, tfa.position`), avoiding an N+1 ā€” called from [track](track.md)'s `ListForUser`/`GetByDedupKey`/`ListTracksFeaturing`, never from `GetByID`/`ListByIDs`/playlist-track joins. `ReplaceFeaturedArtists` verifies track ownership (`SELECT EXISTS ... WHERE id = $1 AND user_id = $2`) before deleting and rewriting the set in one transaction, so the backfill can't cross tenants. `ListTracksFeaturing` joins `tracks`ā†”`track_featured_artists`ā†”`featured_artists` on `fa.identity_key = $2`, ordered newest-first.

Handler (`adapters/handler/featured_artist_handler.go`): `FeaturedArtistHandler` mounts under the track routes (`addRoutes`, called from `TrackHandler.Routes`) as `GET /featuring` (name/mbid/deezer_id query params, precedence mbid ā†’ deezer_id ā†’ name via `FeaturedArtistForQuery`; 400 when none given) and `POST /featured-backfill` ā€” synchronous, no job queue, because the library is small enough that a per-user backfill completes within a request.
