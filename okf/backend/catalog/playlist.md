`Playlist` (`domain/playlist.go`) is identified by the wrapped-UUID `PlaylistId` and holds an ordered `[]PlaylistTrack{TrackId, Position}`. Its methods are the *only* way to mutate membership — never raw slice manipulation: `AddTrack` appends at the next position and returns the domain's `ErrTrackAlreadyInPlaylist` (a `CodedError`, HTTP 409) on a duplicate; `RemoveTrack` deletes and renumbers every subsequent track's `Position` in place, returning `false` (not an error) when the track wasn't a member; `Reorder` requires the exact same track-id set as already present (length mismatch or an unknown id both return `*ValidationError`) and rebuilds `Tracks` with fresh 0..N-1 positions. `NewPlaylist`/`Rename` validate the name (non-empty, ≤100 chars) via `validatePlaylistName`, also `*ValidationError`. Every handler routes these through `httputil.HandleServiceError` — one coded-error vocabulary for the whole context.

`PlaylistSummary` (`TrackCount`, `PreviewArtworkURLs`) and `PlaylistWithSummary` are read-side projection types, deliberately separate from `Playlist` so the write model and its behaviour methods stay free of query artifacts. `PreviewArtworkURLs(tracks []*Track)` (Go-side selection, used on the `Get` handler path where tracks are already loaded) and the SQL projection in `ListForUser` are two independent implementations of the same rule — up to `PreviewArtworkLimit` (4) distinct artwork URLs, position-ordered — sharing the one `PreviewArtworkLimit` constant so the cap itself can't drift, though the two selection algorithms are maintained separately.

Two services, not one — the old bundled doc's "PlaylistService" was actually always two structs:
- **`PlaylistLifecycleService`** (`playlists.go`): `Create`/`List`/`Get`/`Delete`/`Rename`. `List`/`Get` read straight off the repo's projections (`ListForUser` returns `[]PlaylistWithSummary`; `Get` calls `GetWithTracks`). `Rename` re-fetches via `GetByID` (for its `PlaylistSummary`), calls the aggregate's `Rename`, persists via `Update`, and publishes `playlist_renamed`. `Create`/`Delete` publish `playlist_created`/`playlist_deleted`.
- **`PlaylistMembershipService`** (`playlist_membership.go`): `AddTrack`/`RemoveTrack`/`Reorder`. All three follow the same shape — `GetWithTracks` (not `GetByID`; the aggregate's methods need the current membership list loaded) → aggregate operation → persist → publish (`track_added_to_playlist`/`track_removed_from_playlist`/`playlist_reordered`). `RemoveTrack` goes *through* the aggregate rather than straight to the repo specifically so `Playlist.RemoveTrack` stays the single authority for the contiguous-position invariant, matching `Reorder`'s shape. An `AIDEV-NOTE` in `AddTrack` documents a narrow non-transactional race: a concurrent track delete between the `GetWithTracks` read and the repo write can append a now-missing track; accepted pre-launch as a soft failure (reconciled on next read / client retry), not corruption.

Port (`ports/playlist_repo.go`): `PlaylistRepository` is one flat interface — a former `PlaylistStore`+`PlaylistTrackMutator` ISP split was collapsed because no consumer ever used a narrow half (a documented decision in the port's own comment; re-split at the consumer the day one wants less, since Go interfaces are satisfied implicitly and that move costs nothing).

Adapter (`adapters/persistence/playlist_repo.go`): `PgxPlaylistRepository`. `ListForUser` computes `track_count` and `preview_artwork` (an `array_agg` subquery, position-ordered, capped at `PreviewArtworkLimit`) in the same query as the playlist rows — one round-trip for the whole playlists screen. `GetByID` projects `track_count` too (via the same `playlistTrackCountSubquery` constant), so single-playlist responses like rename report the real count without loading track rows. `GetWithTracks` joins `playlist_tracks`↔`tracks` reusing the track package's `trackScanDest()` helper (see [track](track.md)) so the join scans the exact same column set as a bare track read. `RemoveTrack` and track-deletion (in `track_repo.go`, see [track](track.md)) both call the shared `renumberPlaylistPositions` — one definition of the contiguous-position collapse (`ROW_NUMBER() OVER (ORDER BY position) - 1`) so it can't drift between the two write paths. `ReorderTracks` batches its per-track position updates via `pgx.Batch`.

Handler (`adapters/handler/playlist_handler.go`): `PlaylistHandler` holds both services (`lifecycle`, `membership`) and routes `POST /`, `GET /`, `GET /{playlistId}`, `PATCH /{playlistId}`, `DELETE /{playlistId}`, `POST /{playlistId}/tracks`, `DELETE /{playlistId}/tracks/{trackId}`, `PATCH /{playlistId}/tracks/reorder` — all auth-gated. The list endpoint reads track count and preview artwork straight off the `ListForUser` projections with no per-playlist follow-up query; the rename response reports the `GetByID` track-count projection since `playlist.Tracks` is never loaded on that path.
