`Track` (`domain/track.go`) is the catalog context's primary aggregate root, identified by the wrapped-UUID `TrackId` and owned by a `shared.UserId`. It carries title/artist/album plus optional metadata (year, genre, track_number, album_artist, isrc, duration), an ordered `FeaturedArtists []FeaturedArtist` of guest credits (see [featured-artists](featured-artists.md)), and an `AcquisitionStatus` enum (`AcquisitionPending`/`Ready`/`Failed`, zero-value = pending). The status/`AudioRef`/`FailureReason` invariant is enforced only through aggregate methods — `MarkReady(audioRef)` (requires a non-empty ref), `MarkFailed(reason)` (requires a reason, clears the ref), `RevertToPending()` (clears both), `IsStreamable()` (`Ready` AND a non-nil ref) — never by direct field mutation. `NewTrack` validates title/artist (empty either returns `*ValidationError`, part of the catalog-wide `CodedError` vocabulary) and computes a `DedupKey`.

`DedupKey` (`domain/dedup.go`) is `lowercase|NFKC-normalized|alphanumeric-stripped` `title|artist|album`, joined with `|`. It backs the persistence adapter's `ON CONFLICT (user_id, dedup_key) DO NOTHING RETURNING id` upsert (`track_repo.go` `Add`), so re-saving the same track by metadata is idempotent: a conflict falls back to `GetByDedupKey` instead of a race-prone read-then-write, and `TrackRepository.Add` returns `(stored, created, err)` so the caller always gets a usable row either way. A fresh insert runs inside the same transaction as the track's featured-artist links, so track and credits commit atomically.

Services (`service/`):
- **`AddTrackService`** (`add_track.go`) validates via `domain.NewTrack`, copies the request's featured credits onto the track before `Add`, and — only when the row was freshly created (`created == true`) — publishes `track_added_to_library` and schedules acquisition via the injected `ports.AcquisitionScheduler` (a one-method port the acquisition context implements; catalog never imports acquisition). The event embeds the *full* wire-shaped track via `trackAddedPayload` so a receiving client can insert the row directly instead of refetching; the payload marshals `service.TrackDTO` — the same struct the handler serializes as `TrackResponse` (`TrackResponse = service.TrackDTO` type alias) — so event and HTTP response cannot drift. The legacy `track_id` key is retained alongside the canonical `id` for older clients. A narrow `trackAdder` interface (just `Add`) is injected rather than the full `ports.TrackRepository`, so the service's dependency surface matches what it actually calls.
- **`DeleteTrackService`** (`delete_track.go`) deletes the row (`trackRepo.Delete` returns `(deleted, audioRef, err)`; a repo-side transaction also strips the track from any playlists and renumbers their positions — see [playlist](playlist.md)) then best-effort deletes the audio file via `ports.AudioStore.Delete`, logging (not failing) on an orphaned file.
- **`SetTrackNumberService`** (`set_track_number.go`) backs persist-as-you-browse: the client derives a track's real album position from the album tracklist and writes it back. Validates `trackNumber > 0` (else `*ValidationError`), then delegates to the repo's fill-only `SetTrackNumber` (`WHERE track_number IS NULL`) — so re-running never clobbers a real value, and `updated == false` (track already numbered, or not found) is a non-error for an idempotent backfill.
- `ListTracksService`/`GetTrackStatusService` are plain reads, not covered further here.

Adapter (`adapters/persistence/track_repo.go`): `PgxTrackRepository` shares a `trackColumns`/`trackColumnsPrefixed` canonical 17-column list and a `trackScanDest()` scan/construct helper, reused by playlist-track joins (`playlist_repo.go`) and the featured-artist joins (`featured_artist_repo.go`) so a schema change is a single-line edit. `GetByID`/`ListByIDs` deliberately do **not** load featured credits (every current caller of those two only needs status/audio-ref fields); `ListForUser`/`GetByDedupKey`/`ListTracksFeaturing` do, via a shared batch `loadFeaturedForTracks` (avoids N+1). `ListByIDs` (`WHERE user_id = $1 AND id = ANY($2)`) is the batch read behind audio-URL presigning (see [audio-storage](audio-storage.md)) — unknown/foreign ids are simply absent, order is not guaranteed. `Delete` runs in a transaction: it first finds and clears the track's `playlist_tracks` rows (renumbering each affected playlist via the shared `renumberPlaylistPositions`, see [playlist](playlist.md)), then deletes the track row, returning its `audio_ref` for the service's best-effort file cleanup.

Handler (`adapters/handler/track_handler.go`): `TrackHandler` routes (`GET /`, `POST /`, `GET /{trackId}/status`, `PATCH /{trackId}/track-number`, `DELETE /{trackId}`, plus featured-artist routes mounted via `FeaturedArtistHandler.addRoutes`), all auth-gated via `auth.RequireUserID`. `TrackResponse` is a type alias of `service.TrackDTO` — the wire shape is owned by the service layer for the reason above. Validation lives in the domain (`NewTrack` returns a coded 400); the handler does no duplicated pre-checks and routes every service error through `httputil.HandleServiceError`.
