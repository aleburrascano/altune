Acquisition is a customer of the catalog context: it consumes `catalog/domain.Track` and its `MarkReady`/`MarkFailed`/`RevertToPending` invariants rather than owning its own aggregate. There is no `domain/` folder — `AudioCandidate` lives in `ports/ports.go` and doubles as the pipeline's candidate type (same bounded context, same shape — a service-side twin would only duplicate its fields); the pipeline-scoped types (`TrackRef`, `AcquisitionContext`) live in `service/pipeline.go`. This context is process/orchestration-heavy, not aggregate-heavy.

**Pipeline** (`service/pipeline.go`): a `Step` interface (`Name`, `Execute`, `Rollback`) run in order by `RunPipeline`, which rolls back completed steps (reverse order, 30s timeout) on failure and wraps the failing step in a `StepError{Step, Err}`. `AcquireTrackAudioService.buildSteps` (`service/acquire.go`) assembles: `SearchStep` → `SelectStep` → `DownloadStep` → `TagStep` → `StoreStep` → `UpdateTrackStep`.

- **Search** (`step_search.go`): builds several query variants (ISRC, title+artist, +album, +"audio") and calls `AudioSearcher.Search`, deduping the merged results by URL. The yt-dlp adapter (`adapters/ytdlp/searcher.go`) fans each query to two search engines (`ytsearch5:` YouTube, `scsearch5:` SoundCloud), merges/dedupes by URL, and only fails if every engine errors.
- **Select** (`step_select.go` + `service/matching.go`): `rankCandidates` scores every candidate with `identityScore` (title/artist fuzzy match via `textnorm.TokenSortRatio`, penalizing title-only matches), gates out anything below `identityMin` (60), splits survivors into Topic-channel (`isTopicChannel`) vs other, and orders Topic-channel first (then artist-match, then feature-match via `extractFeaturedArtists`/`featureMatch`, then identity), other by identity/feature/metadata-rank.
- **Download** (`step_download.go`): walks the ranked list (capped at `maxVerifyAttempts=4`), downloading each candidate and applying two per-candidate gates before accepting it. Duration gate: when an `AudioProber` is wired and the track has an expected duration, downloads whose probed duration falls outside `durationWithinTolerance` (15s or 7% slack) are rejected. Decode gate: whenever a prober is wired (independent of expected duration), `ValidateDecodable` must pass — a candidate can carry a valid container and the right duration over corrupt samples that no player can decode (the shipped-m4a defect), so undecodable downloads are discarded and the walk falls through to the next candidate; if every candidate fails, the step errors and the Track is never stored. `Rollback` deletes the temp file.
- **Tag** (`step_tag.go` + `adapters/id3/tagger.go`): the step hands the track's metadata to the `AudioTagger` port and swallows (logs) any failure — tagging must never fail the pipeline; without a tagger wired it is a no-op. The id3 adapter writes ID3v2.4 tags (title/artist/album/year/track#/album-artist/genre) via `bogem/id3v2` — but only for `.mp3` files. ID3v2 is an MP3-only convention that prepends a tag block at byte 0; on any other container (e.g. m4a/MP4, which must start with `ftyp` and whose sample-offset table the shift invalidates) that corrupts the file, so the adapter skips non-MP3 files entirely — they carry their metadata in the DB.
- **Store** (`step_store.go`): builds a sanitized `userId/artist/album/title.<ext>` audio ref — the extension is derived from the downloaded temp file itself (falling back to `.mp3`), so the ref always names what the download actually produced; hardcoding the format here is how refs lied about their bytes during the m4a era — and calls `AudioWriter.Store`; rollback deletes the orphaned object. When a prober is wired (`buildSteps` always passes `WithStoreProber(s.audioProber)`, but that's nil — and the check skipped — unless `WithAudioProber` was configured), it re-runs `ValidateDecodable` on the final temp file before persisting — the last gate after download and tagging, catching corruption introduced by any earlier step (a format-mismatched tagger, a truncated write) before it reaches the library; unlike the tag step, this failure does fail the pipeline.
- **UpdateTrack** (`step_update_track.go`): calls `Track.MarkReady(audioRef)` + `SetDuration` and persists; rollback reverts to pending.

**Ports** (`ports/ports.go`): `AudioSearcher` (search+download), `AudioProber` (`ProbeDuration` for true length plus `ValidateDecodable` for playable samples — metadata-only probing can't catch valid-header/corrupt-data files), `AudioTagger` (`Tag(path, TrackTags)` — implementations decide per-container applicability, so skipping an unsupported container is not an error; implemented by the id3 adapter), `AudioWriter` (store-side subset of catalog's `AudioStore`), `TrackRepository` (narrowed get/update). The prober adapter (`adapters/ytdlp/prober.go`, `FfprobeProber`) resolves both ffprobe and ffmpeg binaries near yt-dlp's `--ffmpeg-location` dir (falling back to PATH): duration via ffprobe JSON (30s timeout), decodability via `ffmpeg -v error -i file -f null -` (90s timeout) where a non-zero exit means undecodable (error carries the first stderr line, capped at 200 chars). Both probes fail open — if the binary is missing or times out, verification is skipped rather than blocking acquisition on an unavailable validator.

**Events** (`acquire.go`, `service/scheduler.go`): the service holds an optional `events.Publisher` (`internal/shared/events`, injected via `WithAcquireEvents`; all publishes are nil-guarded for eval/test paths). `Execute` publishes `track_acquisition_started` (`track_id`) before the pipeline runs — the server-authoritative "it's acquiring now" signal the client seeds its download UI from and uses to flip a re-acquired ready/failed Track back to pending, instead of depending on the optimistic save or the poll — then `track_acquisition_completed` (`track_id`, `audio_ref`) or `track_acquisition_failed` (`track_id`, client-safe `reason`). The scheduler's per-job `schedulerJobReporter` holds its own publisher (injected via `WithSchedulerEvents` — not reached through the service) and additionally publishes `track_acquisition_progress` (`track_id`, `stage`) on each stage transition, so the client can show "Finding source… / Downloading… / Finishing up…" live alongside the operator-console job record.

**Scheduling** (`service/scheduler.go` + `service/joblog.go`): `BackgroundAcquisitionScheduler` owns job execution — goroutines gated by a semaphore channel, in-flight dedup per track via `sync.Map.LoadOrStore`, panic recovery, succeeded/failed/in-flight counters — while the operator-console job telemetry lives in `jobLog`, a single-mutex type holding the queued/running job map plus a bounded ring of recent terminal outcomes (failures included, carrying their reason). Failed jobs ride on that `recent` ring with their `reason` — there is no separate failure list; the console reads the failures straight off `recent`. `Status()` combines the counters with a `jobLog` snapshot into `AcquisitionStatus` — in-memory only, reset on restart, no persistence. `Shutdown` cancels and drains via `WaitGroup` with a timeout.

**Retry** (`service/retry_admission.go`, `adapters/handler/retry_handler.go`): `RetryAdmission.Admit(track)` owns the whole admission policy — only tracks in `AcquisitionFailed` state (`ErrRetryNotFailed`), at most one retry per track per 60s cooldown (`ErrRetryCooldown`; in-memory map, opportunistically pruned). `RetryHandler` maps those errors to 409 and 429 respectively; the policy lives service-side so a second retry entry point cannot skip half of it.

`failureReason` (`acquire.go`) maps internal step errors to a small, stable, client-safe vocabulary ("no matching audio found", "audio download failed", "audio storage failed", "audio acquisition cancelled", generic fallback "audio acquisition failed") by matching on the structured `StepError` — RunPipeline's only other error shape is the "pipeline cancelled" prefix — with the full error chain logged, never stored on the track or returned over the wire.
