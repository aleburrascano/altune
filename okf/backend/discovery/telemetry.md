`domain.InteractionEvent` (`domain/events.go`) is the single append-only envelope for every behavioral event: `EventType` (search_performed, results_shown, result_clicked, play, skip, completed, library_add, wrong_album — closed vocabulary, zero-value `EventTypeUnknown` sentinel), `SearchId` (the keystone join key back to the originating search), `EventId` (client-minted idempotency key for the label-critical outbox tier), `ClientOccurredAt` vs. server `OccurredAt`, and a JSONB `Payload` for schema-free growth. Persisted in `discovery_events` (see [discovery-events-table](../../data/discovery-events-table.md)).

Two write paths feed it. `emitSearchEvent` (`service/telemetry.go`) builds the `search_performed` payload — `tail_noise_top5`, the exploration flag, the top-10 shown slate — synchronously on the request path, then hands the actual append to `s.launchBackground(parentCtx, "telemetry.emit", fn)`: a shared helper (`service/search.go`) that now owns the fire-and-forget mechanics (`context.WithoutCancel`, `bgWg.Add`/`Done`, panic recovery logged under `search.v2.background_panic`) previously hand-rolled per call site, and is reused by other async work (e.g. vocabulary ingest, see [vocabulary](vocabulary.md)). Inside the closure, `emitSearchEvent` opens its own `emitTimeout` context before constructing the `InteractionEvent` and calling `ports.EventStore.Append`. `RecordEventService` (`service/record_event.go`) is the inbound use case for client-emitted events, and is the trust boundary (2026-07-23 hardening): it accepts only `EventType.ClientSubmittable()` types (result_clicked, play, skip, completed, library_add, wrong_album — server-reserved `search_performed`/`results_shown` are rejected with 400 so clients cannot forge search metrics), and `validatePayloadTypes` rejects mistyped well-known payload keys (`dwell_ms`/`tail_noise_top5` must be numbers, `zero_result` boolean, `result_signature`/`session_id` strings) so a malformed value can never reach the aggregates' casts.

`PgxEventStore` (`adapters/persistence/event_repo.go`) is the sole adapter satisfying four read/write ports (`EventStore`, `EventQuery`, `BehavioralSignalStore`, `BehavioralLabelStore`) over one `discovery_events` table. Its `SatisfactionSignals` query nets +1 per play/completed and −1 per skip with dwell under the 20s short-dwell threshold (Kim WSDM 2014), grouped by `result_signature`; every JSONB cast in the aggregate queries (here and in the metrics rollup) is `jsonb_typeof`-guarded so one poisoned row skips instead of throwing 22P02 for the whole 30-day window, a skip without a numeric `dwell_ms` counts 0 (not −1), and each user's contribution is capped at ±3 per signature (`perUserSignalCap`) so replayed events cannot pump a result's score. `EventQuery.AbandonedSearches` joins on a `session_id` carried in the JSONB payload to find no-click searches reformulated within 60s — a query-grain coverage gap (Joachims), consumed by `CoverageSignalAService` alongside `ZeroResultQueries`/`NonZeroNoClickQueries` (see [eval-harness](eval-harness.md)), not an `EventConsumer` (that seam is per-`result_signature`, not per-query).

`ports.EventConsumer` (`ports/ports_telemetry.go`) is the Strategy seam: `SatisfactionConsumer` (`service/behavioral_signals.go`) is the first implementation. `Service.RefreshBehavioralScores` recomputes the score map off the request path (a background ticker via `StartBehavioralRefresh`) and atomically swaps it into an `atomic.Pointer`; the search path only ever reads the published snapshot (`Service.BehavioralScoresSnapshot`, exported so the Mission Control re-run ranks with the same behavioral input — structure-audit F1, 2026-07-16), gated behind `behavioralRanking` — a new signal (pogo-sticking, corpus labels) is a new `EventConsumer`, never a pipeline rewrite. Feeds the offline eval loop (see [eval-harness](eval-harness.md)).
