`Merge` (merge.go) collapses per-provider result groups into deduped `Entity` values (a `domain.SearchResult` plus per-provider `BestRank`, feeding Layer 3's RRF tiebreak — see [ranking](ranking.md)). The identity keys it branches on are TYPED fields on `SearchResult` (`ISRC`, `MBID`, `Xref`), not `Extras` entries — compile-checked producer→consumer links, the same rationale as the typed `Popularity`; `completenessOf`'s scoring likewise reads typed `Duration` and `Album` fields directly, not `Extras["duration"]`/an `Extras["album"]` string lookup. `sameEntity` decides identity strictly by tier, strongest first: ISRC exact match (`EntityResolutionISRC`) → shared MBID (`EntityResolutionMBID`, and a *mismatched* MBID is a hard "not the same" even if titles match) → cross-provider identity bridge (`bridgeMatch`/`EntityResolutionBridge`, via a stamped `Xref` that must be present on at least one side, see [identity-artwork](identity-artwork.md)) → bare canonical-name match for artists, or exact canonical title+subtitle for tracks/albums (both `EntityResolutionNone`). There is deliberately no fuzzy threshold or version-marker vocabulary — `textnorm` canonical normalization is the one structural decision defining "same title."

The **ambiguous-artist merge gate** is the one correctness fix layered on top: `ambiguousArtistNames` flags any normalized artist name for which MusicBrainz surfaced ≥2 distinct MBIDs (the "Che" problem — same-name, different humans). It flattens `perProvider` and delegates to `ambiguousArtistNamesFlat`, the shared flat-slice core — the same function `CollapseArtistDuplicates` (diversity.go) calls directly against its already-merged results, so both phases agree on exactly the same ambiguous-name set instead of maintaining parallel logic that could drift. `Merge`'s loop refuses a bare-name merge for those names (identifier/bridge tiers still merge freely); `CollapseArtistDuplicates` is likewise mbid-aware, keying ambiguous names by `name\x00mbid` so distinct identities stay as separate result cards. `mergeInto` folds the loser into the more-complete (`completenessOf`) canonical result, unions `Sources` (deduped by provider+externalID), takes max popularity, coalesces the typed metadata fields (canonical wins when set, else the other side fills the gap — `firstNonEmpty`/`firstNonZero`), and stamps `Confidence` (High for identifier/bridge tiers, Medium for multi-source name matches, Low otherwise — display-only per ADR-0007).

`ConsensusService` (consensus.go) is the detail-screen counterpart: every album provider is an equal source (`ConsensusProvider`, gathered via the generic `FanOutConsensus[T]`), clustered categorically by exact canonical title (no TokenSortRatio threshold) into `ConsensusAlbum` verdicts (confirmed/unconfirmed/rejected). `applyMBAuthority` anchors the union on MusicBrainz's bulk release-group discography when MB resolves the artist: confirmed-by-MB albums are kept, everything else is **rejected** as same-name contamination — replacing a capped, timeout-prone per-album probe. (This MB-veto is the legacy path; the rebuilt discography core — see below — replaces it with corroboration-based keeping.) `ConsensusService.NameGroups` exposes the raw by-name provider albums (one group per responder, un-clustered, un-vetoed) as the by-name completeness feed for that rebuilt core.

**Discography V2 (`DISCOGRAPHY_V2`, docs/discovery-detail-pipeline.md §6).** The rebuilt detail/discography core, replacing the lossy Merge→consensus(MB-veto)→hideBare path. Three pure functions carry the correctness: `MergeReleases` (`release_merge.go`) — field-level **best-of** across every provider that has a release, never the old replace-by-`completenessOf` (which ignored ReleaseDate/Year/TrackCount and dropped them); `KeepRelease`/`FilterKept` (`release_keep.go`) — keep on positive evidence (own-id fan-out, a strong identifier UPC/MBID/ISRC, or ≥2-provider corroboration), so an incomplete MB discography can't purge a real release yet a single-source name-only namesake is still dropped — MB becomes a vote, not a veto; `NormalizeRecordType`/`BucketDiscography` (`release_bucket.go`) — folds the per-provider record_type signals (plus a one-track⇒single rule) into a reliable album/single/EP bucket. `GetArtistContentService.v2Albums`/`v2TopTracks` (`get_artist_content_v2.go`) wire them behind `WithDiscographyV2` on the identity-first path: the id fan-out (IDVerified, queried by id — no blind SoundCloud name-resolve) plus `NameGroups` (unverified) feed `MergeReleases`→`FilterKept`, then record_type/year normalize and order. The per-artist cache is now a pluggable `ports.NameKeyedCache[[]ConsensusAlbum]` (`WithConsensusCache`) rather than an in-process map: unset, `NewConsensusService` defaults to a `noopConsensusCache` (every call recomputes, still correct); the composition root wires a Redis-backed adapter sharing `DefaultConsensusCacheTTL` (6h, exported for that wiring). It still short-circuits the whole fan-out+MB pass, and a timeout-truncated result is never cached. `sortChronological` no longer sorts on `Album.Year` directly — it shares `albumReleaseSortKey` (get_artist_content.go) with `sortAlbumsByReleaseDateDesc` so the artist-detail response's discography and album list agree on "newest first."

`applyArtistDisambiguation` (disambiguation.go) fills empty artist subtitles ("American rapper") from a pre-resolved `extras["disambiguation"]` or, budget-permitting (`disambigMaxLookups = 3`, `disambigTimeout = 2s` — MB is rate-limited to ~1 req/s), a live `ResolveArtistIdentity` MB call, so only the top few displayed same-name artists pay the live-lookup cost.
