The artist **detail** path is a separate pipeline from search (see [scatter-gather](scatter-gather.md)/[merge-dedup](merge-dedup.md)): a detail-open asks for one known artist's full discography and top tracks, so instead of ranking a query it fans out across content providers **by the artist's own id per provider** and merges best-of. `GetArtistContentService` (`get_artist_content.go`) exposes `GetAlbums`/`GetTopTracks`. Both run the identity-first V2 path **whenever an identity store is wired** (`s.identityStore != nil` — production always wires it when a pool exists; the former `DETAIL_IDENTITY_FIRST`/`DISCOGRAPHY_V2` flags were removed 2026-07-23, see [app-wiring](../app-wiring.md)), and fall back to a single-provider path (fetch the seed provider directly, `dedupAlbums`, then the by-name `ConsensusService` union) only when there is no store or the V2 fan-out returns empty. `GetAlbumTracks` (a third endpoint, `GetAlbumTracksService` in `get_album_tracks.go`) resolves one album's tracklist natively per provider (Spotify/Apple/SoundCloud/Deezer/iTunes), with an **artist-guarded** Deezer title-search fallback that returns empty rather than a same-titled album by a different artist.

**Identity resolution** (`detail_identity.go`): `resolveArtistIdentity` expands one `(provider, externalID)` into a `ResolvedArtistIdentity{MBID, ProviderIDs}` by reading the durable [IdentityStore](identity-artwork.md) — keyed on stable ids, never names. The seed id is always preserved, so the result is never narrower than the input even when no cross-provider bridge exists yet (the common case for underground artists MusicBrainz doesn't url-relate — V2 still runs on the seed alone, id-verified). `providerContentID` maps the identity to the id each provider is actually queried by, encoding the two API-imposed exceptions in one place: Last.fm keys artist content on the **MBID** (its "id" is an ambiguous name), and Apple Music reuses the shared **iTunes** id (the bridge only emits the `itunes` key). `resolveArtistIDByName` is a SoundCloud-only fallback (MusicBrainz never bridges SC) — V2 deliberately passes an empty name to disable it, so the id fan-out is queried by id only (contamination-safe). `fanOutByIdentity` (`get_artist_content.go`) runs every provider that has a resolved id concurrently, returning one result group per responder — since 2026-07-23 in a **fixed canonical provider order** (`orderedProviderNames`: deezer, applemusic, spotify, itunes, soundcloud, lastfm, rest alphabetical), because Go map iteration made the `MergeReleases` cluster incumbent random per request (displayed year/title flicker); the fan-out is bounded by `detailFanOutTimeout` (10s) and provider errors are debug-logged instead of silently discarded. `FilterCohesive`'s equal-size component tie also breaks deterministically (lexicographically smallest member).

**The V2 cores** (`get_artist_content_v2.go` orchestrating four pure functions) replace an earlier lossy Merge→consensus(MB-veto)→hideBare path. `v2Albums`/`v2TopTracks` build tagged `ReleaseGroup`s from the id fan-out (all `IDVerified`), then: `MergeReleases` (`release_merge.go`) does field-level **best-of** across every provider that returned a release (a cover, year, or track-count from any source fills the gap — never the old replace-by-completeness that dropped `ReleaseDate`/`Year`/`TrackCount`); `FilterKept` (`release_keep.go`) keeps a merged cluster **iff `IDVerified`** — provenance of the query is the only sound signal for ambiguous names (a namesake's release carries its own valid MBID, and two by-name providers can be the same wrong artist), so by-name groups may enrich an id-verified cluster but a cluster with no id-verified source is dropped; `NormalizeRecordType`/`BucketDiscography` (`release_bucket.go`) fold per-provider `record_type` signals plus a **one-track ⇒ single** rule into reliable album/single/EP buckets. Albums additionally normalize a numeric year from `ReleaseDate` and sort newest-first via `albumReleaseSortKey` (shared with the fallback so both agree on order).

**Two identity-verification guards** clean the fracture where MusicBrainz has a *wrong* url-relation that fuses two same-name artists (the "Che" case: the rapper's MBID url-related to a different Che's Deezer id). `FilterGroupsByMBAnchor` (`release_verify.go`) checks each id-fanout provider group's titles against the MBID's own MusicBrainz release-group set and **drops** a group whose overlap is below threshold (`mbVerifyMinOverlap=4` or `mbVerifyMinRatio=0.25`, only when MB knows ≥`mbAnchorMinReleaseGroups=5` and the group returned ≥`mbVerifyMinTitles=4` — fail-open otherwise); live-measured, a mis-bridged same-name artist shares ~8% of the real artist's release-groups vs 60%+ for the real one. This is applied to `v2Albums` only (album titles are release-groups); `v2TopTracks` relies instead on `FilterCohesive` (`release_cohesion.go`), which unions releases into connected components by cross-provider co-occurrence and drops single-source islands that corroborate with nothing — a soul-Che top-track from one wrong provider id forms its own island and vanishes, while the real artist's multi-source tracks survive. The **permanent** fix is upstream of all this: verify-on-persist ([scatter-gather](scatter-gather.md), `identity_verify.go`) drops a mis-bridged streaming edge before it is stored, so a clean identity never reaches this path — these read-time guards are the belt-and-suspenders. The client detail feature that consumes these endpoints is [detail-feature](../../mobile/detail-feature.md). This path has its own offline regression gate — the `detail` mode of the [eval-harness](eval-harness.md), which feeds seeded *fractured* identities through the real service and asserts the read-time guards keep contamination at 0 (alongside recall and metadata coverage).
