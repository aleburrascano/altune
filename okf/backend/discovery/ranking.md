`Rank`/`rankWith` (rank.go) orders merged `Entity` values (see [merge-dedup](merge-dedup.md)) by continuous, parameter-free relevance — no relevance bands, popularity-dominance window, kind tiers, or intent contract. The experiment-gated inputs (demote predicate, behavioral score map, prominence flag) travel in one `rankConfig` struct whose zero value is the production default — a new experiment is one field, not another positional parameter and wrapper. Pass 1 applies eligibility gates: `sharesQueryWord` (drops results sharing no content token with the query) and `hasBrowseableSource` (artist/album need a Deezer source; tracks always pass). Pass 2 scores each eligible entity via `scored{relevance, behavioral, prominence, pop, rrf, multi, demoted}` and `rankLess` sorts by: demotion flag → relevance → prominence → behavioral score → popularity → multi-source → RRF (`rrfK=60`) → stable subtitle/title tiebreak. The prominence rung compares **unconditionally** (2026-07-23): the old kind-difference gate made `rankLess` a non-transitive comparator (comparison cycles → formally unspecified sort output for tied-relevance mixed-kind groups, the exact bare-name case the rung serves); lawfulness is pinned by a strict-weak-ordering property test. `sharesQueryWord` also no longer force-gates single-character queries (an empty query token set with a non-empty normalized query passes everything). Both passes live in `rankScored`, the shared scoring core: `rankWith` calls it and drops to `[]SearchResult`, while the exported `RankExplain` (see [scatter-gather](scatter-gather.md)) calls the *same* `rankScored` and keeps each result's `scored` fields as a `ScoredResult` — so the Mission Control rank explainer shows the exact math production ranked on and can never drift from it. The behavioral map is keyed by `domain.ResultSignature` — the single definition the wire mapper also emits, so stored scores always join the live list. `pop` and the tail-identity check below read the typed `SearchResult.Popularity`/`.Album` fields directly — rank.go no longer goes through a `popularityOf`/`stringExtra` indirection to get them, mirroring diversity.go's `CollapseArtistDuplicates`, which made the same switch off the extras map.

Relevance itself (`rank_relevance.go`, `idfWeightedCoverage`) is `Σ_qtoken IDF(token)·bestFuzzyMatch(token, title+subtitle) / Σ_qtoken IDF(token)`: IDF weight (`queryTokenRarity` in rank.go, `rarity = 1 - documentFrequency/N` over the eligible candidate set) comes from the data, not a tuned constant, so a repeated "artist" token in an "artist title" query weighs ~0 while the rare "song" token weighs ~1; `bestTokenSimilarity` is a normalized Levenshtein ratio, continuous with no cutoff. When every query token is ubiquitous (IDF can't distinguish), it falls back to `symmetricSimilarity` — the published token-sort ratio — so an exact title still beats a superset title. This replaced the pre-rebuild `distinguishingBoostWeight = 0.35` tuned constant.

Two eval-gated experimental rungs, both default-off and inert unless their composition-root env flag is set: `isLowConfidenceTail` (tail-noise demotion — flags single-source SoundCloud/Last.fm results with no ISRC/MBID/album, read straight off the typed `SearchResult` fields, as low-confidence tail, sorting below every corroborated result; `TailNoiseInTopK` is the tracked quality signal) and `prominenceOf` (cross-kind prominence — log-compressed Deezer fan count/track rank, read from the typed `SearchResult.FanCount`/`ProviderRank` fields, so a famous bare-name artist rises above a same-name track; since the 2026-07-23 comparator-lawfulness fix it also orders same-kind relevance ties ahead of popularity, both being popularity-family signals).

`diversity.go`'s `EnforceDiversity` (cap `maxPerArtistInTop=3` within `diversityWindow=10`) and `CollapseArtistDuplicates` (fold same-name artist duplicates, keying by MBID when the name is ambiguous per `ambiguousArtistNamesFlat` — the same flat-slice core `Merge` calls, see [merge-dedup](merge-dedup.md); an MBID-less entity under an ambiguous name keys by its first source's provider+id so two distinct no-MBID namesakes never collapse into one card) are explicitly **product policy**, not the query-fit constants the rebuild purged — they're exempt from the zero-arbitrary-constants doctrine and validated via the `cmd/discoveryeval -mode diversity` harness (which uses `rankPipelineNoReshape` as its baseline — see [eval-harness](eval-harness.md)).
