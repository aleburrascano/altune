The load-bearing principle (per `ARCHITECTURE.md`): everything merges by identity, never by bare name, because same-name entities (the "Che" problem) otherwise collide. Two structures carry this in-request and durably.

`domain.ArtistIdentityProfile` (`domain/identity.go`) is an in-flight read-model — MBID, Discogs id, genre cluster, ISRC registrants, MB-confirmed titles — assembled per search from provider signals; `AlbumVerdict` (Confirmed/Contamination/Suspect/Unknown) is the consensus classification an album gets against it (see [merge-dedup](merge-dedup.md)).

The durable counterpart is `entity_identity` (see [entity-identity-table](../../data/entity-identity-table.md)): `ports.IdentityStore` maps `(provider, external_id, kind) → (mbid, xref)`. `PgxIdentityStore` (`adapters/persistence/entity_identity_repo.go`) is the source of truth — `PersistBridges` upserts one row per bridged provider id when MusicBrainz answers a search, with the upsert MERGING `xref` (`old || new`, new wins per key) so edges an earlier pass learned survive a partial re-learn; `mbid` stays last-write-wins (whether a conflicting mbid should instead be rejected is an open question). `LookupByProviderID` reads it back, still degrading any DB error to a miss but debug-logging it (`identity.lookup_failed`) so a real DB problem isn't indistinguishable from a miss. The port also has `Invalidate(kind, provider, externalID)` — Postgres DELETE plus Redis DEL — the purge/remediation surface for excising a bad bridge after the fact; nothing on the search/detail pipeline calls it. `RedisIdentityStore` (`adapters/cache/identity_store_cache.go`) fronts it with a 30-day read/write-through cache, degrading transparently to the inner store when Redis is nil or on a miss (a failed cache-warm `Set` is debug-logged, not swallowed silently) — this is what lets a later MB-absent search still resolve a provider-only result's identity deterministically instead of guessing by name (`service/artwork_fill.go`'s `fillArtworkOne`, tagged `"durable-identity"` in `artwork_path`).

Artwork resolution consumes this identity. The service-side port is `ports.TaggingArtworkResolver` (url + source, name and identity variants) — the single interface the search service and MB enrichment consume; the plain `ports.ArtworkResolver` is the chain-MEMBER interface only (one artwork source), and the chain's members declare optional capabilities via `IdentityArtworkResolver` (fetch-by-bridged-id, never name-searches) and `SourcedArtworkResolver` (names itself for coverage tagging). `ChainedArtworkResolver` (`adapters/providers/artwork_chain.go`) tries identity-only resolvers first via `ResolveWithIdentityTagged` (Discogs-by-id, CAA/Fanart by MBID) — these never name-search — and only falls to name-search resolvers (`ResolveTagged`) when no identity source has the image. The bridged ids ride the typed `SearchResult.Xref` field (stamped pre-merge, or recovered from the durable store). `ports.ArtworkConfidence` (`ArtworkConfidenceIdentity` > `ArtworkConfidenceName` > `None`) grades the result so `RedisArtworkCache` (`adapters/cache/artwork_cache.go`) can refuse to let a weaker name-guess overwrite a proven-identity image, and gives identity hits a 14-day TTL vs. 48h for provisional name hits and a per-kind negative TTL (tracks churn fastest, artists slowest). See [artwork-chain](../../providers/artwork-chain.md) for the ordered provider chain itself.
